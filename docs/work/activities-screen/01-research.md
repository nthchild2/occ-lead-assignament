## Existing patterns to follow

- **Screen-level list with loading/error/empty branching** ‚Üí follow `app/app/(protected)/(tabs)/index.tsx:121-175` (`ListContent`). Same triage order: `isLoading && items.length === 0` ‚Üí skeletons, `error && items.length === 0` ‚Üí `ErrorState`, `!isLoading && !error && items.length === 0` ‚Üí `EmptyState`, else render the list.
- **Zustand store consumption** ‚Üí hook-selector style for reactive fields (`useApplicationsStore((s) => s.items)`), `getState()` for imperative one-off calls (`fetch`, `remove`) ‚Äî mirrors `app/app/(protected)/(tabs)/index.tsx:180-182, 217-255` and the `index.test.tsx:110-120` mocking precedent for both call shapes in the same test.
- **Optimistic remove with inline error surfacing** ‚Üí follow `app/app/(protected)/JobDetail.tsx:302-322` (`handleApply`/`handleToggleFavorite`): `try { await store.getState().remove(id) } catch (err) { setMessage(messageFor(err)) }`. The store already rolls back on failure (`app/store/applications.store.ts:69-76`, `app/store/favorites.store.ts:63-70}`) ‚Äî the screen only needs to catch-and-display, never re-throw further.
- **Reusing `JobCard` with a secondary action** ‚Üí do **not** modify `JobCard.tsx`'s props (`app/core/components/JobCard.tsx:9-12`); its memo comparator (`JobCard.tsx:78-80`) and existing 5-test contract (`JobCard.test.tsx`) are keyed to exactly `{ job, onPress }`. Wrap it in a row `View` alongside a separate `Button` (`variant="danger"` or `"secondary"`, mirrors `JobDetail.tsx:280-285`'s favorite-toggle button) for "Cancelar"/"Quitar" ‚Äî composition over prop-contract change, consistent with `Card`/`Badge`/`JobCard`'s single-responsibility split (MAP.md:34).
- **Avoiding duplication between `applied.tsx` and `favorites.tsx`** ‚Üí both screens need near-identical fetch-on-mount + loading/error/empty/list logic operating on structurally different `items` (`Application[]` vs `Job[]`). Extract a shared internal list component (e.g. `ActivityList`) parameterized over already-normalized `{ id, job }` rows (or a small per-screen adapter that maps `Application ‚Üí { id: jobId, job }` and `Job ‚Üí { id, job: item }` before handing off to the shared component), following the same "extract to keep complexity ‚â§ 10" rationale as `FilterBar`/`ListContent` (`index.tsx:56-175`). Two route files (`applied.tsx`, `favorites.tsx`) stay thin: wire their own store + adapter, delegate rendering to the shared component.
- **List virtualization choice: `FlatList`, not `FlashList`** ‚Üí A5 Decision 2 (`docs/A5 ¬∑ Performance.md:60-62`) frames FlashList optimizations around "the job search screen," explicitly called out as "the highest-traffic surface in the app" ‚Äî not a blanket rule. Applications/favorites lists are fetched in one shot (`fetch()` takes no pagination args ‚Äî confirmed in both stores and services), are per-user and bounded in size, and are not the perf-critical surface A5 targets. `app/core/components/Select.tsx:88` already uses plain `FlatList` for a similar small, non-paginated list. Using `FlatList` here avoids the FlashList jest-mock machinery entirely (`index.test.tsx:15-84`) and keeps these tests simpler (RNTL renders `FlatList` natively).
- **`setActiveJob` for R8** ‚Üí `useJobsStore.getState().setActiveJob(job.id, index)`, same call as `app/app/(protected)/(tabs)/index.tsx:254-256`. For applications, pass `item.job.id` (the embedded `Job`, confirmed at `packages/shared/src/schemas/application.schema.ts:4-8`) and the row's local array index; for favorites, pass `item.id` directly (items are already `Job`). The mismatch between this local index and `jobs.store.jobs` is harmless ‚Äî `isSwipeDisabled` (`JobDetail.tsx:122-129`) already disables swipe whenever the index doesn't resolve to the same job in `jobs.store.jobs`, and `useActiveJob` (`JobDetail.tsx:50-103`) falls back to `jobsService.getById` when the job isn't in the store ‚Äî both already-correct behaviors for a job opened from outside the search list.
- **Test mocking** ‚Üí mock `app/store/applications.store.ts` / `app/store/favorites.store.ts` at the module boundary (`jest.mock('../../../../store/applications.store', ...)`), not `global.fetch` ‚Äî same reasoning as `index.test.tsx:11-20` (store/hook already unit-tested separately).

## Constraints that apply

- `core/` must not import from `app/` ‚Äî `.eslintrc.js:32-44` (A1 architecture boundary). A shared `ActivityList`-style component that only takes props (no store reads) could live in `core/components/`; if it needs route-local adapter logic it should stay route-adjacent instead, per MAP.md:36's guidance that route-lifecycle-coupled components belong under `app/app/`.
- `complexity: ['error', 10]` ‚Äî `.eslintrc.js:26`. Each screen must extract subcomponents (list/row) the way `index.tsx`'s `FilterBar`/`ListContent` do; do not inline all four states (loading/error/empty/list) plus row-rendering in one function body.
- No `any`; `@typescript-eslint/consistent-type-imports` enforced ‚Äî `.eslintrc.js:13,15`.
- No direct store `setState` from components ‚Äî components call store actions (`fetch`/`remove`), never mutate state directly ‚Äî `docs/A4 ¬∑ Quality Strategy.md:486`.
- No inline color/size/spacing literals ‚Äî consume `useTheme()` tokens, per MAP.md:33, matching every existing screen/component read here.
- Accessibility: interactive elements need `accessibilityRole` + label ‚Äî MAP.md:37 (A4 Decision 8); `Card`'s `onPress` wrapper already sets `accessibilityRole="button"` (`Card.tsx:40`) and `JobCard` sets an `accessibilityLabel` (`JobCard.tsx:34`) ‚Äî the new cancel/remove `Button` also needs an explicit accessible label (e.g. via `accessibilityLabel` or a descriptive `label`) distinct from the card's own label so screen readers can distinguish "open detail" from "cancel/remove" on the same row.
- App tests are co-located (`*.test.tsx` next to the screen) ‚Äî MAP.md:60. `jest-expo`; mock stores/services, not `global.fetch`/msw ‚Äî MAP.md:62.

## Risks & unknowns

- **Row layout for the cancel/remove action**: no existing component in this codebase combines a `Card`-based pressable row with a second, independent pressable action inside/beside it (a11y-wise, nesting a `Button`'s `Pressable` inside `Card`'s own `Pressable` wrapper must be avoided ‚Äî `Card.tsx:39-43` wraps its entire content in one `Pressable` when `onPress` is given). The plan should decide: either render `JobCard` **without** its own `onPress` inside a custom outer `Pressable` row (losing `Card`'s built-in tap semantics), or place the cancel/remove `Button` as a sibling **outside** `JobCard`'s `Card` boundary (e.g., `JobCard` for tap-to-open, a separate `Button` below/beside it for cancel) so the two pressables never nest. The latter is the safer default (no nested-Pressable a11y ambiguity) and requires no `JobCard` change ‚Äî flagging for the plan to make explicit, not a blocking ambiguity.
- **Confirmation before cancel/remove**: spec R3/R4 say "immediate feedback" and cite the store's existing optimism ‚Äî no confirmation dialog is mentioned or required. Plan should confirm this is a direct tap-to-remove (no "are you sure?" step), consistent with "immediate feedback" language.
- Both open questions flagged in the spec (`00-spec.md:36-37`) are resolved by this research: `JobCard` reuse via external composition (no prop change), and `Application.job` as the embedded `Job` for `setActiveJob`. Neither requires escalation.

## Handoff to PLAN

- Two route files (`app/app/(protected)/(tabs)/activities/applied.tsx`, `favorites.tsx`) each: fetch on mount from their store (`useApplicationsStore`/`useFavoritesStore`, `app/store/applications.store.ts:11`, `app/store/favorites.store.ts:11`), render via a shared list component to avoid duplicating the loading/error/empty/row logic (mirrors `FilterBar`/`ListContent` extraction at `app/app/(protected)/(tabs)/index.tsx:56-175`).
- Use `FlatList`, not `FlashList` ‚Äî these are low-traffic, non-paginated, per-user lists (A5's FlashList optimizations target only the search screen: `docs/A5 ¬∑ Performance.md:60-62`); precedent at `app/core/components/Select.tsx:88`. This also sidesteps the FlashList jest-mock machinery in tests.
- Reuse `JobCard` unmodified (`app/core/components/JobCard.tsx:9-12`); add a sibling `Button` (not nested inside `JobCard`'s `Card` pressable) for cancel/remove, mirroring `JobDetail.tsx:280-285`'s favorite-toggle button and its catch-and-display error pattern (`JobDetail.tsx:302-322`).
- Loading/error/empty triage order and component choices are fixed by precedent: `isLoading && items.length===0` ‚Üí `JobCardSkeleton` √ó N (`Skeleton.tsx:46`), `error && items.length===0` ‚Üí `ErrorState` (`message={error}`, `onRetry={fetch}`), `!isLoading && !error && items.length===0` ‚Üí `EmptyState` with a screen-specific title/description.
- R8: `useJobsStore.getState().setActiveJob(job.id, index)` ‚Äî for applications pass `item.job.id`/`item.job`; favorites items are already `Job`. No changes needed to `jobs.store.ts`, `JobDetail.tsx`, or `(protected)/_layout.tsx` ‚Äî the existing fallback-fetch and swipe-disable logic already handles jobs opened from outside the search list correctly.
- Tests: co-locate `applied.test.tsx`/`favorites.test.tsx` (and the shared list component's own test, if extracted to a separate file), mocking `applications.store`/`favorites.store` at the module boundary per `index.test.tsx:15-20`'s precedent ‚Äî do not re-test store internals (already covered by `applications.store.test.ts`/`favorites.store.test.ts`).
