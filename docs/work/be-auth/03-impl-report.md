## Deviations from the plan

- **Plan steps 5/6 — line references.** The plan cited `app.ts:20` for the placeholder and `app.ts:28` for `app.listen()`; the actual file had the domain-zone placeholder at lines 19-23 and `app.listen` at line 28. The intent was unambiguous (mount at the `/auth` placeholder; guard the bootstrap), so I applied both edits at the correct current locations. No behavior difference from the plan.
- **Plan step 3 — `verifyToken` return typing.** `jwt.verify` returns `string | JwtPayload` (per the `jsonwebtoken` package types), so `decoded.sub`/`decoded.email` are not directly accessible under strict mode without narrowing. I normalize the verified payload into the internal `JwtPayload` inside `verifyToken` (coercing `sub`/`email` to strings) rather than accessing fields off the raw union. No `any`, no `!`. The user is still derived from the verified payload as planned.
- **Plan step 2 — `expiresIn` typing.** `SignOptions.expiresIn` is the narrow `ms` `StringValue | number` template type (from the `jsonwebtoken` types, imported from the `ms` package), but `env.JWT_EXPIRES_IN` is a plain `string`, which is not assignable to it under strict mode. I build a single typed `SignOptions` object and coerce the value via `env.JWT_EXPIRES_IN as SignOptions['expiresIn']` (a type-level cast to the option's own type — not `any`, not `!`). This was an unavoidable strict-mode necessity the plan didn't call out; it does not change runtime behavior (`jwt.sign` accepts the `'1h'` string as before).
- **R8 self-containment — jest `setupFiles` added (orchestrator-applied resolution).** As delivered, the supertest suites required `JWT_SECRET` to be present in the shell (the config's fail-fast `.parse()` runs at import when `app` is imported), so `pnpm --filter './backend' run test` failed in a clean shell — the R8 tests weren't self-contained. Fixed by adding `backend/jest.setup.ts` (`process.env.JWT_SECRET ||= 'test-secret-…'`, which runs before any test module imports) and wiring it via `"setupFiles"` in `backend/package.json`'s jest block. Backend suite now passes in a clean shell (`env -u JWT_SECRET …` → 5 suites / 24 tests green, no open-handle warning). This is tooling config needed for R8 to actually run; no production code changed.

## Anything left for a follow-up

- **Blacklist persistence.** The blacklist is a single-instance in-memory `Set`, cleared on restart and not shared across instances — documented in-code as sanctioned tech debt (A1 Decision 5, spec "Explicitly out of scope"). Redis/DynamoDB-backed invalidation is a separate ticket, not done here.
- **Jobs / applications / favorites, refresh tokens, real user store, app-side auth store/login screen** — all explicitly out of scope per the spec; not touched. This ticket only provides the reusable `authMiddleware` those domains will import.
