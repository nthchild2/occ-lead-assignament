## Existing patterns to follow

- **Non-persisted Zustand store** → follow `app/store/theme.store.ts:22-30`: plain `create<T>((set) => ({...}))`, no curried `()(persist(...))`. `jobs.store.ts` follows this exact shape (A2 Decision 1 explicitly excludes jobs from persistence).
- **Thin service wrapping the api client** → follow `app/core/services/auth.service.ts:12-23`: one exported function per operation, explicit `auth` boolean passed to `get`/`post`, response schema imported from `@occ/shared`. `jobs.service.ts` exports one function (e.g. `list(filters: JobFilters)`) that builds the query string and calls `get(path, JobListResponseSchema, false)`.
- **Selector-style hook wrapping a store** → follow `app/core/hooks/useTheme.ts:5` for the "thin passthrough" idiom; `useJobs`/`useDebounce` extend this with async orchestration and local state/effects respectively (no existing hook in this codebase yet does that — first of its kind here).
- **Fetch-mock unit testing** → follow `app/core/services/api.test.ts:1-38` and `app/store/auth.store.test.ts:1-34`: mock `global.fetch` per-test with a `mockResponseOnce(status, body)` helper, set `EXPO_PUBLIC_API_BASE_URL` in `beforeAll`, reset `global.fetch = jest.fn()` in `beforeEach`.
- **Plain-store reducer testing (no fetch)** → follow `app/store/theme.store.test.ts:1-19`: call actions directly via `useJobsStore.getState().action(...)`, assert via `useJobsStore.getState()`. No `renderHook` needed for the store's own reducer tests (`appendJobs`/`resetList`/`setFilters`) — only the hooks (`useJobs`, `useDebounce`) need `renderHook`/`act` from `@testing-library/react-native`.

## Constraints that apply

- `get(path, schema, auth?)` takes only a path **string** — no query-object parameter (`app/core/services/api.ts:110-116`). `jobs.service.ts` must build the querystring itself before calling `get`; there is no existing `buildQueryString`/serialization helper anywhere in `app/core/` — **this ticket needs to add one** (either inline in `jobs.service.ts` or as a small local helper), since none currently exists to reuse.
- No `fetch`/`axios` in stores — only in `core/services/` (`docs/MAP.md:46`, `docs/MAP.md:72`). `jobs.store.ts` must contain zero network calls; all fetching happens in `useJobs.ts` via `jobs.service.ts`.
- `core/` must not import from `app/` (`.eslintrc.js:33-42`, rationale A1 Decision 2). `useJobs.ts`/`useDebounce.ts` live in `app/core/hooks/` and must not import anything from `app/app/`.
- No `any` — API types via `z.infer<>` (`.eslintrc.js:13`). Use `Job`, `JobFilters`, `Pagination` types imported from `@occ/shared`, not hand-written interfaces.
- `complexity: ['error', 10]` (`.eslintrc.js:23`) — keep `fetchPage`/`fetchNextPage`/`refetch` and the querystring builder simple; extract helpers if branching grows.
- `react-hooks/rules-of-hooks` and `react-hooks/exhaustive-deps` are linted (`.eslintrc.js:26-27`) — `useDebounce`'s `useEffect` dependency array must be correct (value + delay).
- Only `auth.store` is persisted; all other stores reset on launch (`docs/MAP.md:45`, A2 Decision 1, `docs/A2 · State & Data Strategy.md:40`). `jobs.store.ts` must not use `persist` middleware.
- Test convention: co-located `*.test.ts`, mock `global.fetch`/service module, no msw (`docs/MAP.md:59-61`).

## What NOT to touch

- `app/core/services/api.ts` — the api-client contract (`get`/`post`/etc. signatures) is already implemented and verified by `app-api-client`; this ticket only consumes it, does not modify it.
- `app/store/auth.store.ts` — out of scope; `jobs.store.ts` does not need to read/write auth state (jobs list endpoint is unauthenticated per R1/`auth=false`).
- `packages/shared/src/schemas/job.schema.ts` — schemas are already correct and complete for this ticket's needs (confirmed above); no schema changes required.
- `backend/src/domains/jobs/*` — `be-jobs` is already implemented (per spec's "Explicitly out of scope" and confirmed present on disk); this ticket is app-side only.
- Any screen/UI files under `app/app/` — explicitly out of scope per spec ("Screen UI" belongs to `job-search-screen`); this ticket delivers store/service/hooks only, no rendering.
- `activeJobId`/`activeJobIndex` swipe-threshold/prefetch-trigger _logic_ — this ticket only provides the state and `setActiveJob`/`clearActiveJob` actions; the threshold-triggered `fetchNextPage()` call from swipe events belongs to `job-detail-swipe`.

## Risks & unknowns

- **A2 doc drift (non-blocking, same pattern flagged in `app-auth-store`):**
  - A2's inline `JobsStore.pagination` type (`docs/A2 · State & Data Strategy.md:70`: `{ page: number; total: number; hasNext: boolean }`) is a narrowed 3-field shape that does not match the real `PaginationSchema` from `@occ/shared` (5 fields: `page, limit, total, hasNext, hasPrev`, `packages/shared/src/schemas/job.schema.ts:14-20`). The plan should use the real `Pagination` type from `@occ/shared`, not A2's inline sketch — R2's acceptance criterion says "Store shape matches A2's `JobsStore` interface" but the fields listed there (jobs, filters, pagination, isLoading, error, activeJobId, activeJobIndex) are the part that matters; the exact `pagination` field's shape should come from `@occ/shared`'s `Pagination` type since that's what `appendJobs(jobs, pagination)` in R3 receives directly from `JobListResponseSchema`'s parsed response.
  - A2's illustrative `useJobs` hook (`docs/A2 · State & Data Strategy.md:107-133`) returns only `{ fetchNextPage, refetch }` from the hook, even though it defines `fetchPage` internally and never exposes it. Spec R6 explicitly requires the hook to expose `fetchPage(page)` too. The plan should follow the spec (R6) over A2's stale snippet — expose all three (`fetchPage`, `fetchNextPage`, `refetch`).
  - Neither of these is a blocking ambiguity: the spec's R2/R6 acceptance criteria are precise enough to resolve both discrepancies without human input. Flagging for the plan phase to make the deliberate choice explicit (spec + real shared schema win over the stale A2 snippet), not as an AMBIGUITY ESCALATION.
- **Query-string construction has no existing helper.** Confirmed `get()` only accepts a path string (`app/core/services/api.ts:110-116`) with no query-serialization utility anywhere in `app/core/`. The plan must include a small helper (e.g. `buildQueryString(filters: JobFilters): string`) inside or alongside `jobs.service.ts`. Care needed: `JobFiltersSchema`'s `.default()` values (`sort: 'date_desc'`, `page: 1`, `limit: 20`) only apply when the schema _parses_ input — a plain in-memory `JobFilters` object from the store won't have those defaults auto-filled unless the store's initial state seeds them explicitly (relevant to R2's initial `filters` shape and R4's `resetList()` reset target).
- **Undefined/optional filter fields must be omitted from the querystring**, not serialized as `"undefined"` — the backend router casts `req.query` directly to `JobFilters` (`backend/src/domains/jobs/jobs.router.ts:21`) with its own Zod parse/defaults (per `backend/src/domains/jobs/jobs.schema.ts:13-16`, mirrors `JobFiltersSchema` defaults), so the app-side builder should skip `undefined`/empty values rather than pass literal strings.
- No existing precedent in this codebase for a hook that both calls Zustand actions and performs async fetching with error handling (`useJobs` is the first). `useTheme.ts` is a pure selector with no async logic, so there's no existing "async hook + renderHook test" pattern to copy verbatim — the plan should design this test structure fresh, using `renderHook`/`act` from `@testing-library/react-native` (confirmed available) combined with the `global.fetch` mocking convention from `api.test.ts`/`auth.store.test.ts`.

## Handoff to PLAN

- `get(path, schema, auth?)` is path-string-only (`app/core/services/api.ts:110-116`) — plan must include a small query-string builder in/near `jobs.service.ts`; no existing helper to reuse. Jobs list is unauthenticated, so always pass `auth=false`.
- `jobs.store.ts` must use the plain `create<T>((set) => ({...}))` form like `app/store/theme.store.ts:22-30`, never the curried `persist(...)` form from `app/store/auth.store.ts:18-55` — per A2 Decision 1, jobs is not persisted.
- Use the real `@occ/shared` types (`Job`, `JobFilters`, `Pagination`, `JobListResponse` — `packages/shared/src/schemas/job.schema.ts:46-50`) as the store/service/hook contracts, not A2's inline sketch types, which have drifted (narrower `pagination` shape, `useJobs` missing `fetchPage` in its return — see Risks).
- Follow `app/core/services/auth.service.ts:12-23` for `jobs.service.ts`'s shape (one function, explicit auth flag, shared schema) and `docs/MAP.md`'s no-fetch-in-stores rule — all network calls belong in `jobs.service.ts`, called only from `useJobs.ts`.
- Test plan: store reducers (`appendJobs`/`resetList`/`setFilters`) tested directly via `getState()` like `app/store/theme.store.test.ts`; `useJobs`/`useDebounce` (both new kinds of hook in this codebase — first async/timer hooks) need `renderHook`/`act` from `@testing-library/react-native` (confirmed present, `app/package.json:53`) combined with the `global.fetch`-mock convention from `app/core/services/api.test.ts` and `app/store/auth.store.test.ts`.
- `fetchNextPage`'s in-flight guard uses the store's `isLoading` (already resolved in spec, `docs/work/jobs-store/00-spec.md:39`, consistent with A2's swipe-prefetch snippet at `docs/A2 · State & Data Strategy.md:194`) — no separate lock/ref needed.
- No existing `jobs.store.ts`, `jobs.service.ts`, `useJobs.ts`, or `useDebounce.ts` — all four are new files; `app/core/hooks/` currently only has `useTheme.ts`/`useThemeFonts.ts`.
