## Existing patterns to follow

- **Envelope construction** → never hand-roll `res.json({ error: … })`; call `fail(res, status, code, message)` (`backend/src/lib/response.ts:7-14`). The current inline handler at `backend/src/app.ts:26-28` builds the object by hand — R3 should route through `fail` for consistency.
- **Error logging** → use the exported `pino` `logger` (`backend/src/lib/logger.ts:3`), matching `backend/src/app.ts:25` (`logger.error({ err }, 'Unhandled error')`). No `console.*` (guardrail, `.eslintrc.js:18`).
- **Backend module layering** → per A1 Decision 4 (`docs/A1 · Monorepo Architecture.md:154-161`): `*.schema.ts` = pure types/Zod, no framework imports; middleware may import Express. The taxonomy + `AppError` are framework-free (a `lib/` or domain-agnostic module); the two middlewares import Express types. Config module is framework-free.
- **Zod schema location for shared API shapes** → shared, cross-consumer schemas live in `packages/shared/src/schemas/` (`docs/MAP.md:49-50`). The error _taxonomy_ / `AppError` are backend-internal (thrown server-side, never a client type), so they belong under `backend/src/` — not `packages/shared`. Only the envelope shape (`ApiErrorSchema`) is shared and already exists.
- **Express error-handler signature** → the 4-arg handler with an unused final param uses `_next` (`backend/src/app.ts:24`); `.eslintrc.js:20-25` documents `argsIgnorePattern: '^_'` exists precisely for this. Follow that when extracting into `error.middleware.ts`.
- **`type`-only imports** → `.eslintrc.js:15` enforces `consistent-type-imports`; follow `backend/src/lib/response.ts:1` (`import type { Response } from 'express'`) and `backend/src/app.ts:2` (`import express, { type Express }`).
- **Backend test placement** → tests go in a `__tests__/` directory with `.test.ts` suffix to match `backend/package.json:26` `testMatch: **/__tests__/**/*.test.ts`. NOTE: A4's examples (`docs/A4 · Quality Strategy.md:60,81`) and `docs/MAP.md:59` show _co-located_ `*.test.ts` — those describe the **app** workspace; the backend's actual jest `testMatch` requires `__tests__/`. Trust the config, not the co-location prose, for backend tests. (See Risks.)

## Constraints that apply

- No `any`; API types via `z.infer<>` — `.eslintrc.js:13` (`@typescript-eslint/no-explicit-any: error`). R1's taxonomy must be a union/discriminated type, not `string` (matches R1 acceptance).
- No non-null assertions (`!`) — `.eslintrc.js:14`. Config validation (R5) must narrow types via Zod parsing, not `process.env.JWT_SECRET!`.
- No `console.*` — `.eslintrc.js:18`; use `logger` (also `docs/MAP.md:73`).
- Cyclomatic complexity ≤ 10 — `.eslintrc.js:26`. Keep the error-middleware branching (AppError vs unknown) and the `validate()` factory small.
- Backend security plugin active on `backend/**/*.ts` — `.eslintrc.js:48-67` (warnings): avoid object-injection / non-literal patterns when mapping error `code`→`status`; prefer a typed lookup record over dynamic indexing.
- TypeScript strict mode repo-wide — `tsconfig.base.json:2` (`strict: true`); `noUnusedParameters` (`tsconfig.base.json:7`) means the error handler's unused params need `_` prefixes.
- Response envelope is `{ data }` / `{ error: { code, message } }`, no `ok` field — A1 Decision 6 (`docs/A1 · Monorepo Architecture.md:211-242`), `.github/CLAUDE.md:17`. R3 output must conform to `ApiErrorSchema` (`packages/shared/src/schemas/application.schema.ts:22-27`).
- Services must not import Express; middleware may — A1 Decision 4 (`docs/A1 · Monorepo Architecture.md:163`), `.github/CLAUDE.md:13`. The taxonomy/`AppError`/config must stay Express-free so services can throw `AppError` without pulling in Express.
- Error middleware must be the LAST `app.use` — `backend/src/app.ts:23` ("must be last"); R3 preserves ordering.

## What NOT to touch

- `backend/src/lib/response.ts` — the `success`/`fail` helpers already exist and are correct; the middleware _uses_ `fail`, it does not modify or replace it (spec "Explicitly out of scope").
- `backend/src/app.ts:9-15,36-42` — CORS, `express.json`, `/health`, graceful shutdown are pre-built and out of scope. Only edits allowed in `app.ts`: swap `process.env.PORT` for config (R5) and replace the inline handler (lines 24-29) with `app.use(errorMiddleware)`.
- `packages/shared/**` — `ApiErrorSchema` already exists at `application.schema.ts:22-27`; no schema change needed. Changing shared schemas requires lead review (`.github/CLAUDE.md:50`) and is out of scope.
- `backend/src/domains/**` — no domain router/service/seed in this ticket (spec out-of-scope; ROADMAP `be-jobs`/`be-auth`/`be-apply-fav` own those).
- `backend/src/lib/logger.ts` — pino logger already exists; reuse, don't modify.
- Auth/JWT verification middleware — belongs to `be-auth` (spec out-of-scope); this ticket only defines the `JWT_SECRET`/`JWT_EXPIRES_IN` env vars, it does not consume them.

## Risks & unknowns

- **Backend test directory convention conflict (non-blocking, flag for planner).** `backend/package.json:26` sets `testMatch: **/__tests__/**/*.test.ts`, but A4 (`docs/A4 · Quality Strategy.md:60,81`) and `docs/MAP.md:59` show co-located `*.test.ts`. A co-located backend test would be silently skipped. Plan R8 tests into `backend/src/**/__tests__/*.test.ts` to match the running config. (The prose examples describe the app workspace; not an app/backend contradiction that blocks work.)
- **R6 may already be satisfied.** `.env.example:1-6` already lists `PORT`, `JWT_SECRET`, `JWT_EXPIRES_IN` (plus `NODE_ENV`, `LOG_LEVEL`). R6 acceptance ("lists `PORT`, `JWT_SECRET`, `JWT_EXPIRES_IN` with placeholder values") appears met — planner should treat R6 as verify-only (a no-op or a comment tweak), not new work. The file is currently mode `-rw-------`; no permission change is in scope.
- **`NODE_ENV` typing (R5).** R5 lists `PORT`, `JWT_SECRET`, `JWT_EXPIRES_IN`, `NODE_ENV`. `logger.ts:5` branches on `NODE_ENV === 'development'`; the config's `NODE_ENV` enum should include at least `development`/`production`/`test` so existing logger behavior isn't broken. Not blocking — a planning detail.
- **Taxonomy code set is slightly wider than downstream tickets.** R1 requires `ALREADY_FAVORITED` (used by `be-apply-fav`) and `TOKEN_EXPIRED`/`AUTH_REQUIRED`/`INVALID_CREDENTIALS` (used by `be-auth`). These are defined here but not exercised until later tickets — expected (foundational plumbing), not scope creep.
- No blocking ambiguities. Both open questions in the spec (JWT_EXPIRES_IN default `'1h'`; `validation.middleware.ts` naming) were resolved in `00-spec.md:37-38` and are consistent with A1's `*.middleware.ts` convention.
