## Deviations from the plan

- **Plan step 4 / R6 — added a jest `moduleNameMapper` entry in `app/package.json` for `@react-native-async-storage/async-storage` (not listed in the plan).** Running the new test suite failed at import time with the native module's own guard error (`[@RNC/AsyncStorage]: NativeModule: AsyncStorage is null`), because `auth.store.ts` imports the real native `AsyncStorage` module, which has no JS-side implementation under plain `jest-expo` without a mock wired in. This is the same class of infra gap the plan's risk section flagged for msw/module-registry leakage, just surfacing at a different point (native module resolution, not fetch). Fix: mapped the module to AsyncStorage's own officially-documented Jest mock (`@react-native-async-storage/async-storage/jest/async-storage-mock`, shipped inside the already-declared dependency — no new package added), via the same `moduleNameMapper` mechanism already used for `@occ/shared`. This is infrastructure required for R6 ("Unit tests for the session store") to be runnable at all, not a design change to `auth.store.ts`/`auth.service.ts`; no production code changed as a result.
- **R6 test-genuineness fix (orchestrator-applied after a VERIFY FAIL).** The first VERIFY pass found the R4 test ("onUnauthorized clears the session") called `hydrate()` to trigger the 401, but `hydrate()` has its own independent `catch { clearSession() }` (R5's logic) — a mutation to `onUnauthorized` (making it a no-op) left that test passing, proving it never actually exercised the R4 wiring. Fixed by rewriting the test to trigger the 401 via `login()` instead: `login()`'s failure path only propagates `ApiError` and never calls `clearSession()` itself, so if the session still clears, it can only be `configureApi`'s `onUnauthorized` firing — cleanly isolated from R5.
  - **A second, deeper genuineness defect surfaced while fixing the first**: the test file's `beforeEach` was _redundantly re-wiring_ `configureApi({ onUnauthorized: () => useAuthStore.getState().clearSession() })` before every test (a defensive measure against cross-file module-registry leakage). This masked bugs in `auth.store.ts`'s own module-load wiring entirely — even with the module's real `onUnauthorized` mutated to a no-op, the suite still passed, because `beforeEach` silently re-installed the correct closure regardless. Fixed by deleting the redundant `beforeEach` re-wiring; the suite now exercises only the wiring that happens once, as a side effect of `import { useAuthStore } from './auth.store'` at the top of the test file — the actual production code path.
  - **Verified genuine, this time correctly:** re-ran the same mutation (`onUnauthorized: () => {}` in `auth.store.ts`) — the R4 test now fails (`expect(received).toBeNull() // Received: "tok-1"`) while all 14 other tests still pass (no collateral coupling). Reverted the mutation; confirmed `auth.store.ts` is byte-identical to the implementer's original. Full suite green again: 15/15, `tsc` clean, `eslint` 0 problems on the changed file.
- All other plan steps (1-4) were implemented exactly as specified. No other deviations.
