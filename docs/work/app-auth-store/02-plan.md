## Approach

Two new files, built bottom-up so the tree stays type-checkable: `app/core/services/auth.service.ts` first (a thin wrapper around the already-shipped `get`/`post` helpers in `app/core/services/api.ts`, typed entirely off `@occ/shared`'s `User`/`LoginRequest`/`LoginResponse`/`MeResponse`), then `app/store/auth.store.ts` (a Zustand store using the curried `create<T>()(persist(...))` v4 form, since `persist` requires currying unlike the plain `create` in `theme.store.ts`). The store exposes `login`, `logout`, `hydrate`, and a `clearSession` primitive that both `logout` and the injected `onUnauthorized` handler share — this satisfies the spec's requirement that `onUnauthorized` never re-enter the network (avoiding a 401-retry loop) while keeping one source of truth for "wipe the session." `configureApi` is wired once at module load, mirroring `theme.store.ts`'s `Appearance.addChangeListener` idiom (`useAuthStore.getState()` read at call time, not captured at store creation). `hydrate()` is an explicit, directly-testable store action calling `me()` — rejected alternative: relying on `persist`'s `onRehydrateStorage` callback alone, which fires automatically post-storage-read and cannot be invoked/asserted deterministically in a unit test (R6 requires exactly that). Tests mock `global.fetch` directly (per `docs/MAP.md`'s convention and `api.test.ts`'s established pattern), not the service module, so the store tests exercise the real service→api-client wire path end to end, same rigor as `api.test.ts`.

## Risks & rollback

- **`persist` curried API mismatch.** If `create<AuthStore>()(persist(...))`'s generic inference doesn't line up cleanly (a known zustand v4 TS friction point), the fallback is to type the `set`/`get` callback params explicitly rather than relying on inference. Rollback: the change is isolated to one new file; delete/rewrite `auth.store.ts` without touching any other module.
- **`LogoutResponseSchema` shape mismatch with the real backend.** Since no shared schema exists, the local minimal schema (`z.object({ data: z.unknown() })`) is a guess at the envelope shape. If `be-auth`'s actual `/auth/logout` response differs (e.g. no `data` key, or a `204 No Content` with an empty body), `schema.parse` in `api.ts:103` will throw on a successful call, and `logout()`'s `try/catch` will treat it as a failure it swallows — session is still cleared (best-effort), so this fails safe but silently. Rollback/fix: narrow the schema further once `be-auth`'s live response is inspected (out of scope for this ticket's verify gate, since `be-auth` is a separate, already-implemented ticket per the spec's dependency list).
- **Test flakiness from module-level `configureApi` wiring.** Because `configureApi` is called once at `auth.store.ts` module load (change 3), and `api.test.ts` also calls `configureApi({})` in its own `beforeEach` to reset config, importing both `auth.store.ts` and `api.ts` in the same Jest module registry could cause one test file's wiring to leak into another if Jest doesn't reset modules between files. Mitigation: `auth.store.test.ts` should call `configureApi({})` or explicitly re-wire in its own `beforeEach`/`afterEach` if cross-file leakage is observed during verify; Jest's default per-file module isolation should prevent this in practice (each test file gets a fresh module registry), so no code change is anticipated — flagged for the implementer to watch during verify.
- **Rollback strategy overall:** both new files are additive (no existing file is edited), so if verify fails, the safe rollback is deleting `app/core/services/auth.service.ts` and `app/store/auth.store.ts` (and their test file) with no ripple to `api.ts`, `theme.store.ts`, or `@occ/shared`.
