## Existing patterns to follow

- **Zustand store (singleton, actions via `set`)** → follow `app/store/theme.store.ts:22-30`. One store per domain (`docs/MAP.md:44-47`); components call actions, never raw setters.
- **Module-level side-effect wiring reading fresh state via `getState()`** → follow `app/store/theme.store.ts:32-35` (`Appearance.addChangeListener` calls `useThemeStore.getState()` at invocation time, not a stale closure). Use the same idiom for the `configureApi({ getToken: () => useAuthStore.getState().token, onUnauthorized: () => ... })` call resolved in the spec's open questions.
- **Service wrapping the api client with Zod response schemas** → follow the shape of `get`/`post` calls already used in `app/core/services/api.test.ts:41-49` (`get('/me', MeResponseSchema)`); `auth.service.ts` should be a thin set of functions (`login`, `logout`, `me`) each calling `post`/`get`/`del` from `app/core/services/api.ts` with the matching schema from `@occ/shared`, returning `z.infer<>`-typed data — no `fetch` in the service (MAP.md:72).
- **Fetch/service mocking in tests, not msw** → follow `app/core/services/api.test.ts:15-38` (`global.fetch = jest.fn()`, `mockResponseOnce` helper, reset in `beforeEach`). Per `docs/MAP.md:61`, for `auth.store` tests the lighter option is mocking the **`auth.service` module** (`jest.mock('../core/services/auth.service')`) since the store's contract (R6) is about state transitions, not HTTP wire format — the wire format is already covered by `api.test.ts`. Cite this convention in the plan; MAP.md explicitly allows either "mock `global.fetch` or the service module."
- **Pure-store unit test style (no RTL render)** → follow `app/store/theme.store.test.ts:1-19` — call actions via `useThemeStore.getState().action()`, assert via `useThemeStore.getState()`.
- **Zod schema reuse, no hand-written duplicate types** → follow `packages/shared/src/schemas/auth.schema.ts:24-27`; import `User`, `LoginRequest`, etc. as `z.infer<>` types from `@occ/shared` (A1 Decision 1, `docs/MAP.md:50`).

## Constraints that apply

- No `fetch`/`axios` in stores or components — only in `core/services/` — `docs/MAP.md:72`; enforced structurally (the store must call `auth.service.ts`, which calls `app/core/services/api.ts`).
- `core/` must not import from `app/` (routes) — `.eslintrc.js:32-40`. This does not block `store/` → `core/services/`, but it is the documented reason `api.ts` uses dependency injection (`configureApi`) instead of importing `auth.store` directly (`app/core/services/api.ts:5-8`) — the new store must preserve this one-way direction (`store` imports `core`, never the reverse).
- No `any` — API types must come from `z.infer<>` (`.eslintrc.js:13`, `docs/MAP.md:71`). Use `LoginRequest`, `LoginResponse`, `MeResponse`, `User` from `@occ/shared`.
- `no-console: error` (`.eslintrc.js:10`) — no `console.*` in the new files.
- `complexity: ['error', 10]` (`.eslintrc.js:22`) — keep `hydrate()` simple; extract a helper if the rehydrate-then-validate flow grows branchy.
- Only `auth.store` is persisted; all other stores are not (A2 Decision 1, `docs/A2 · State & Data Strategy.md:40`; `docs/MAP.md:45`). This ticket touches no other store.
- Response envelope on the wire is `{ data }` / `{ error: { code, message } }`, already handled by `api.ts`'s `mapError`/`ApiError` — the service/store consume already-unwrapped `z.infer<>` data or a thrown `ApiError`, never raw envelopes.
- Zustand version installed is `4.5.7` (satisfies `app/package.json:38`'s `^4.5.0`). The `persist` middleware (and `createJSONStorage`) is available via the `zustand/middleware` subpath export declared in the installed package's `exports` map — `import { persist, createJSONStorage } from 'zustand/middleware'`. No version bump needed.
- Zustand v4 `persist` middleware requires the curried `create<T>()(persist(...))` call form (the double-parens currying) when combined with middleware, per the `StateCreator` generic signature — this differs from the plain `create<T>((set) => ({...}))` form used in `theme.store.ts` (which has no middleware). The plan must account for this signature difference.
- Zustand v4 `persist` middleware's `PersistOptions` type exposes `onRehydrateStorage?: (state: S) => ((state?: S, error?: unknown) => void) | void` — a factory called once when rehydration starts, returning an optional callback invoked when rehydration finishes (with the rehydrated state or an error). Separately, the store instance created by `persist` exposes a `.persist.rehydrate(): Promise<void>` method (accessible as `useAuthStore.persist.rehydrate()`) that manually re-triggers a rehydration on demand. These are two different mechanisms: `onRehydrateStorage` is a passive hook fired automatically after the initial async storage read completes; `.persist.rehydrate()` is an imperative re-trigger. Per the spec's resolved ambiguity, the plan should pick one concrete mechanism for `hydrate()` (R5) — e.g., an explicit `hydrate()` **action** on the store (called once from app startup, out of scope here) that calls `authService.me()` and updates/clears state, decoupled from (but possibly triggered after) the `onRehydrateStorage` callback signals rehydration is complete. Because `onRehydrateStorage` only fires once automatically per store creation and is awkward to unit-test deterministically, an explicit `hydrate()` store action (testable directly via `useAuthStore.getState().hydrate()`) is the pattern that best fits R5's acceptance criterion ("calling `hydrate()` calls `me()`") and R6's testability requirement — this is a plan-level decision, not asserted as fact here.

## Risks & unknowns

- **A2 Decision 1's illustrative `auth.store` snippet (`docs/A2 · State & Data Strategy.md:44-61`) has `login(token: string, user: User)`** — i.e., the store receives already-obtained credentials — **while spec R2 requires `login(email, password)`**, meaning the _store's_ `login` action itself calls `auth.service.login(email, password)` internally and only then sets `token`/`user`. This is not a blocking contradiction: A2 predates the service-layer split (`app-api-client`/`app-auth-store` weren't separate tickets when A2 was written) and the spec is the authoritative, more recent source for this ticket. The plan should implement R2's `login(email, password)` signature (which internally calls the service) and treat A2's snippet as directionally illustrative only, not a literal contract. No escalation needed — flagging for the planner's awareness.
- **`POST /auth/logout` HTTP verb.** The spec (R1) says `auth.service.ts` wraps `POST /auth/logout`. `app/core/services/api.ts` exposes `post` (and `del`), so `post('/auth/logout', ...)` is the correct call — no ambiguity, just noting the api client has no bespoke "logout" helper; it's a plain `post` call. The response schema for logout is not defined in `packages/shared/src/schemas/auth.schema.ts` (only `LoginRequestSchema`/`LoginResponseSchema`/`MeResponseSchema` exist — no `LogoutResponseSchema`). The plan must decide: either logout's response is ignored (fire-and-forget, e.g. `post('/auth/logout', z.void())` or a minimal inline schema) or a schema needs adding to `@occ/shared` (out of scope per spec's "no schema changes" implied by the dependency list only naming three schemas). Recommend the plan use a minimal local Zod schema (e.g. `z.object({}).passthrough()` or reuse an existing generic) rather than touching `@occ/shared`, since the spec explicitly scopes `@occ/shared` deps to `LoginRequestSchema`, `LoginResponseSchema`, `MeResponseSchema`, `User` only.
- **`onUnauthorized`/`getToken` wiring call site.** The spec's resolved open question says this happens "at module load time in `auth.store.ts` (or a small init function invoked once)." This is a plan-level implementation detail (call `configureApi` either as a side effect at the bottom of `auth.store.ts`, mirroring `theme.store.ts:32-35`'s module-level `Appearance.addChangeListener` call, or inside an explicit `initAuth()` export). Both are consistent with the codebase's existing singleton-with-module-level-wiring pattern; no blocking ambiguity, but the plan must pick one concretely.
- **`hydrate()` mechanism choice** (`onRehydrateStorage` vs. explicit action vs. `.persist.rehydrate()`) is a plan-level decision — see "Constraints that apply" above for the full mechanism comparison. Not blocking; flagged for the planner to resolve explicitly in `02-plan.md`.

No AMBIGUITY ESCALATION is required — all open questions above are resolvable at the plan level using patterns already present in the codebase.
