## Existing patterns to follow

- **Themed placeholder screen** тЖТ follow `app/core/components/EmptyState.tsx:1-56`: `useTheme()` for colors/spacing/type, inline `style={{ ... }}` objects reading `theme.x`, no `StyleSheet.create`, no literals.
- **Reading theme without a Context provider** тЖТ `useTheme()` (`app/core/hooks/useTheme.ts:5`) reads a Zustand store (`app/store/theme.store.ts:22`), not React Context тАФ so no `<ThemeProvider>` needs to be added to the root layout's provider stack; the theme is already globally available the moment the store module loads.
- **Gating render on font load** тЖТ `useThemeFonts()` (`app/core/hooks/useThemeFonts.ts:6-9`) returns a plain boolean; root layout should hold `if (!loaded) return null` (or a minimal loading view) before rendering `<Slot />`/providers' children, matching the spec's "renders nothing (or a loading state) until resolves `true`."
- **Store action shape for reset-on-event** тЖТ `applications.store.ts:78-80` / `favorites.store.ts:72-74`'s `reset()` тАФ plain synchronous setter, safe to call from an effect or a logout handler without awaiting.
- **Module-scope service wiring pattern** тЖТ `auth.store.ts:60-63`'s comment explicitly documents the "closures read `getState()` at call time" idiom (mirrored from `theme.store.ts:32-35`'s `Appearance.addChangeListener`) тАФ relevant if the guard needs to subscribe to `auth.store` outside of a component (e.g., a `useAuthStore.subscribe(...)` for reacting to a live 401-triggered `clearSession()`), rather than only checking `token` once in a `useEffect` on mount.

## Risks & unknowns

- **A2 Decision 3's code snippet is stale against the real `auth.store` API** (`docs/A2 ┬╖ State & Data Strategy.md:140-166`): it shows the guard living in `app/_layout.tsx` (root layout, not `(protected)/_layout.tsx`) and calling `authService.me()` directly + a synchronous-looking `logout()`, never referencing `hydrate()` at all. Neither A2 nor A3's snippets exactly match the built `hydrate()`-based API. **Resolution for PLAN:** treat A3's _placement_ (`(protected)/_layout.tsx`, not root) as authoritative per spec R3's explicit wording ("`(protected)/_layout.tsx` guards the subtree тАж calls `hydrate()` and redirects to login on failure"), and treat the spec's R3 acceptance criterion (calls `hydrate()`, not raw `authService.me()`) as authoritative over both docs' code snippets, since `hydrate()` already encapsulates the exact try/catch+clearSession behavior both snippets hand-roll. This is not a blocking ambiguity тАФ the spec is unambiguous on this point; the architecture docs are just showing older/simplified illustrative code that predates `hydrate()`'s addition to the store.
- **A2 Decision 3's summary flow (`docs/A2 ┬╖ State & Data Strategy.md:236-241`) lists a bare "jobs.store reset" step, but `jobs.store.ts` (`app/store/jobs.store.ts:4-19`) has no `reset()` action тАФ only `resetList()` (line 14, resets `jobs`/`pagination`/`isLoading`/`error`, not `activeJobId`/`filters`).** R9 in the spec only lists `applications.store`/`favorites.store`'s `reset()` as `should`-priority, and does not mention `jobs.store` at all. **Resolution:** follow the spec's R9 wording exactly (only `applications`/`favorites` reset calls are in scope for this ticket); flag that `jobs.store`'s reset story is either intentionally out of scope here or a gap for a later ticket тАФ not this ticket's problem to solve, since spec R9 doesn't ask for it.
- **Provider nesting order is not pinned by any doc with a literal ordering list.** Resolved from Expo Router's documented convention (not a node_modules citation, per instructions): `GestureHandlerRootView` must be the outermost wrapper (gesture-handler's own documented requirement тАФ it must wrap the entire app so all gesture responders register against it), `SafeAreaProvider` next (safe-area-context's own convention тАФ provide insets before anything that reads them, incl. screens and the bottom sheet), then `BottomSheetModalProvider` (gorhom/bottom-sheet's documented requirement тАФ must wrap any tree using `useBottomSheetModal`/rendering `BottomSheetModal`), with the font-loading gate (`useThemeFonts()`) short-circuiting render _before_ the route `<Slot />`/`<Stack />` mounts (but the gate check itself can sit inside or outside the provider stack тАФ outside is simpler and avoids mounting gesture/safe-area machinery before fonts resolve). Suggested order top-to-bottom: `GestureHandlerRootView` тЖТ `SafeAreaProvider` тЖТ font-gate (`if (!loaded) return null`) тЖТ `BottomSheetModalProvider` тЖТ `<Slot />` (root `_layout.tsx`, no route groups yet at this level тАФ group-level guards live in `(auth)/_layout.tsx` and `(protected)/_layout.tsx` per A1/A3's tree). This is a reasoned convention resolution, not a fabricated citation тАФ PLAN should state it as a design decision, not attribute it to a specific doc line.
- **Expo Router route-group/layout-nesting conventions** (not cited to node_modules, per instructions): parenthesized folders (`(auth)`, `(protected)`, `(tabs)`) are Expo Router's "route groups" тАФ they organize routes without adding a URL segment. Each group with its own `_layout.tsx` gets its own navigator (e.g., a `Stack` or `Slot`), and nested groups compose (root `_layout.tsx` renders a `Slot`/`Stack` containing both `(auth)` and `(protected)` as children; `(protected)/_layout.tsx` itself renders a `Slot` wrapping `(tabs)`; `(tabs)/_layout.tsx` is a `Tabs` navigator; `(tabs)/activities/_layout.tsx` is a nested top-tab navigator, e.g. `Tabs` or `MaterialTopTabs`). Redirects between groups use `<Redirect href="..." />` or `router.replace(...)` from `expo-router`, evaluated during a layout's render/effect based on store state тАФ this is the mechanism both R3 and R4 rely on.
