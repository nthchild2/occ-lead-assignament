## Approach

Build the Expo Router tree fresh under `app/app/` exactly as documented in `docs/A1 · Monorepo Architecture.md:34-47`, ordered root → `(auth)` → `(protected)` → `(tabs)` → `activities`, so every intermediate step still type-checks (each layer's `_layout.tsx` only needs `expo-router` primitives and already-existing stores/hooks, never a sibling file that doesn't exist yet). All guard logic calls the real `auth.store` API (`hydrate()`, `clearSession()`, `token`) per spec R3's explicit wording, not the stale `authService.me()`/raw `logout()` snippets in A2/A3 — research flags those docs as illustrative and pre-dating `hydrate()`. `BottomSheetModalProvider` is mounted once, at the root layout (not duplicated at `(protected)`), because R2 already requires it in the root provider stack and R8 only requires it to wrap `(protected)`'s subtree — nesting it twice would be redundant and root-level coverage is a superset. R9 (should) is wired at the single most defensible call site: a small `logout` handler colocated with `(protected)/_layout.tsx`'s guard effect, since that is the one place in this ticket's scope that already observes `auth.store` and controls navigation — not scattered across future screen tickets that don't exist yet. The rejected alternative was putting the provider stack for `BottomSheetModalProvider` inside `(protected)/_layout.tsx` only (matching A3's Decision 2 literally) — rejected because R2's acceptance criterion already lists `BottomSheetModalProvider` as a root-layout provider, and splitting it would mean `(auth)` screens can't use bottom sheets later without a second provider instance; mounting once at the root satisfies both R2 and R8 with no duplication and no behavioral difference (the `(auth)` subtree simply doesn't use it yet).

### Per-file responsibility (design-level, no full code)

- **`app/app/_layout.tsx`** — Owns the provider stack and the font-load gate. Nesting: `GestureHandlerRootView` → `SafeAreaProvider` → `if (!useThemeFonts()) return null` → `BottomSheetModalProvider` → `<Slot />`. Knows nothing about auth or route groups; `<Slot />` renders whichever top-level group (`(auth)` or `(protected)`) Expo Router resolves. This satisfies R2 (provider order + font gate) and R8 (single `BottomSheetModalProvider`, no sheet content).
- **`app/app/(auth)/_layout.tsx`** — Reads `token`/`user` from `useAuthStore` (no `hydrate()` call here — that only lives in `(protected)`, per spec's division of responsibility: `(auth)` only checks locally-known session state to bounce an already-logged-in user away from `/login`; it does not itself validate the token against the server). If a session looks valid, `<Redirect href="/(protected)/(tabs)" />`; otherwise `<Slot />` renders `login.tsx`. Satisfies R4.
- **`app/app/(auth)/login.tsx`** — Minimal themed placeholder (`<View style={{ backgroundColor: theme.colors... }}><Text style={{ color: theme.colors... }}>Login</Text></View>`), no form fields (out of scope). Satisfies R1.
- **`app/app/(protected)/_layout.tsx`** — The guard. On mount: if `token` is absent, `<Redirect href="/(auth)/login" />` immediately (no network call needed — nothing to hydrate). If `token` is present, a `useEffect` calls `hydrate()` once, with a local `checking` state (`'checking'` while the hydrate promise is in flight, `'ready'` once resolved) so the subtree doesn't flash-render before hydration settles; if hydration fails (i.e. `hydrate()` resolves but `token`/`user` end up null — `hydrate()`'s internal `clearSession()` path per research), redirect to `/(auth)/login`; once `checking === 'ready'` and a session is confirmed, render `<Slot />`. This is also where the R9 logout helper is defined (`handleLogout`, not yet wired to a trigger UI since no logout button exists in this ticket's placeholders — it is exported/available for later screen tickets to call, e.g. via context or simply documented as the pattern to follow): calls `auth.store.logout()`, then `applications.store.reset()` and `favorites.store.reset()` (not `jobs.store` — no `reset()` action exists there per research; out of scope). The subsequent `token === null` is what the guard's own redirect-on-no-token branch picks up to navigate to login — no separate `router.replace()` call is needed inside `handleLogout` itself, avoiding a duplicate navigation path. Satisfies R3 and R9.
- **`app/app/(protected)/(tabs)/_layout.tsx`** — Expo Router `Tabs` navigator, two `Tabs.Screen` entries: `index` (title "Búsqueda") and `activities` (title "Actividades"), `screenOptions` pulling colors from `useTheme()` (active/inactive tint, background) — no inline hex literals. Satisfies R5.
- **`app/app/(protected)/(tabs)/index.tsx`** — Themed placeholder standing in for the job-search screen. Satisfies R1.
- **`app/app/(protected)/(tabs)/activities/_layout.tsx`** — Nested tab navigator (Expo Router `Tabs` reused for the top-tab/segmented look, `screenOptions` themed) with two screens: `applied` (title "Postuladas") and `favorites` (title "Favoritos"). Satisfies R6.
- **`app/app/(protected)/(tabs)/activities/applied.tsx`** and **`favorites.tsx`** — Themed placeholders standing in for the future applications/favorites lists. Satisfy R1.

## Tests to add or update

**N/A for this ticket, by design — no test files are planned.** Justification:

- `docs/A4 · Quality Strategy.md:33,37` is explicit: screen/component snapshot tests are added "in the same PR that completes the feature, after the developer considers the screen done" and are "not added to screens under active development." Every file this ticket creates is a placeholder explicitly superseded by later screen tickets (`login-screen`, `job-search-screen`, `activities-screen` per spec's "Explicitly out of scope") — none of them are a "done" screen, so A4's own gate says no snapshot belongs here yet.
- The spec's requirements ledger (R1–R9) contains no acceptance criterion phrased as a unit-testable assertion about route/layout behavior in isolation (e.g., no "a test verifies X" wording) — these are integration-level navigator/provider wiring, not pure functions or store logic, and Expo Router layouts are conventionally verified by the app running (R7's own acceptance criterion says "verified via Expo's linking config, not a live device test," reinforcing that this ticket's verification is config/structural, not test-suite based).
- Guard _logic_ that is meaningfully unit-testable (`hydrate()`'s success/failure branching, `logout()`'s `clearSession()` behavior) already has its test coverage in `app-auth-store`'s ticket (`app/store/auth.store.ts` is consumed here, not modified) — re-testing it through a layout-render harness would duplicate that coverage without adding signal, and `.eslintrc.js`'s `complexity: 10` cap already pushed the guard's design toward small, previously-tested primitives rather than new inline logic worth isolating.
- This is a deliberate scope decision, not a silent skip: if a reviewer wants route-level smoke tests (e.g., "does `(protected)/_layout` redirect when `token` is null"), that would be a new `R`-id to add to the ledger and re-plan against, not something to add unilaterally here.

## Risks & rollback

- **Guard hydration race / flash-render:** if `(protected)/_layout.tsx`'s `checking` state is mismanaged, an authenticated-looking user could see a flash of `<Slot />` content before `hydrate()` resolves, or an unauthenticated user could briefly see protected placeholders. Mitigation: the local `checking` state must gate `<Slot />` rendering (render `null` or a lightweight loading view while `checking === 'checking'`), matching the same "render nothing until resolved" convention as the root font-gate. Rollback: revert `app/app/(protected)/_layout.tsx` to the redirect-only version (no `hydrate()` call) if the stateful version introduces a regression verify catches — R3's `must` bar is met by the redirect alone; `hydrate()` is the refinement.
- **Duplicate navigation on logout:** if `handleLogout` (R9) both calls `router.replace('/(auth)/login')` directly _and_ relies on the guard's `token === null` branch to redirect, Expo Router could receive two competing navigation calls in one tick. Mitigation: `handleLogout` only calls the store actions (`logout()`, `applications.store.reset()`, `favorites.store.reset()`); it does not call `router.replace()` itself — the guard's existing no-token branch is the single source of navigation truth. Rollback: if this single-path approach still misbehaves in verify, fall back to an explicit `router.replace()` inside `handleLogout` and remove reliance on the guard's reactive branch — a small, isolated change.
- **`complexity: 10` breach in `(protected)/_layout.tsx`:** this file accumulates guard + hydration + logout/reset wiring (R3 + R9), the most logic-dense file in this ticket. Mitigation: extract the hydration effect and/or `handleLogout` into a local helper function within the same file (not a new `app/core` hook, since `core/` must stay navigation-agnostic per `.eslintrc.js`'s `import/no-restricted-paths`) if the guard function itself approaches the limit. Rollback: if extraction still doesn't fit, defer R9's wiring out of this file into a follow-up ticket (R9 is `should`, not `must` — dropping it entirely from this ticket is an acceptable fallback that still passes verify).
- **Typed routes (`experiments.typedRoutes: true`) rejecting string literals:** `<Redirect href="/(protected)/(tabs)" />` / `"/(auth)/login"` string literals must match Expo Router's generated route types once `app/app/` exists. Mitigation: run typecheck immediately after creating each layout (per the ordered build sequence) so a route-string typo is caught at the step that introduced it, not at the end. Rollback: fix the literal (e.g., trailing segment mismatch) — no structural change needed, this is a typo-class risk only.
- **`app.json` scheme accidentally reverted:** since research found `scheme: "occ"` already set, the risk is an editor/tool touching `app.json` while working on the plugins list. Mitigation: change #1 is a read-only verification step, no edit tool should target `app.json` in this ticket. Rollback: trivial — re-add the one-line `scheme` field if a diff shows it was dropped.
