### Jest-config fix — independent verification (the critical check for this run)

The orchestrator's narrative in `03-impl-report.md` §Deviations was **not** taken on trust. Independently reproduced and falsified:

1. **Static config check.** `app/package.json`'s jest block has `"resolver": "react-native-worklets/jest/resolver"` and `"react-native-reanimated": "react-native-reanimated/mock"` under `moduleNameMapper` (keyed `"^react-native-reanimated$"`). No `setupFiles` entry exists anywhere in the block. Confirmed by direct read of `app/package.json:40-51`.
2. **`git diff app/package.json` scope check.** Diff is exactly one line added (`"resolver": "react-native-worklets/jest/resolver",`) plus the pre-existing `moduleNameMapper` entry for `react-native-reanimated` — both isolated to the `jest` block. Nothing else in the file (dependencies, scripts) was touched.
3. **Resolver file existence and content.** Opened `node_modules/.pnpm/react-native-worklets@0.8.3_.../node_modules/react-native-worklets/jest/resolver.js` directly (not trusted from the report). Confirmed it exists and does exactly what's claimed: it strips `.native.*` extension candidates specifically when the request or basedir involves `react-native-worklets`, which is the documented workaround for the native-init error under Jest.
4. **Resolver-removal regression test (load-bearing proof).** Temporarily removed only the `"resolver"` line (kept `moduleNameMapper`) and reran `pnpm --filter './app' run test`: **1 suite failed** (`app/(auth)/login.test.tsx`, the only suite that renders `Button`, which imports `react-native-reanimated`) with the exact `WorkletsError: [Worklets] Native part of Worklets doesn't seem to be initialized` described in the report; the other 11 suites (which don't render components) stayed green. This is a more precise result than the report's "all 12 suites" framing for _this specific_ config permutation, but is consistent with the resolver being load-bearing for any suite that touches `react-native-reanimated`.
5. **Full original-bug reproduction.** Went further: removed the `moduleNameMapper` reanimated entry too and restored the implementer's original (broken) `"setupFiles": ["react-native-reanimated/mock"]`, then reran the suite. Result: **all 12 suites failed** with the identical `WorkletsError`, 0/63 tests ran — because `setupFiles` eagerly `require()`s the mock for every test file regardless of whether that suite touches reanimated, while `moduleNameMapper`/`resolver` only intervene when a module is actually imported. This exactly reproduces and confirms the report's "broke all 12 suites" claim for the config the implementer actually committed.
6. **Restoration.** Restored `app/package.json` from a pre-experiment backup, reran the suite: **12/12 suites, 63/63 tests green**, and `git status`/`git diff --stat` confirmed the working tree returned to exactly the original one-line diff (no residual changes from the experiment).

Conclusion: the resolver + moduleNameMapper fix is genuinely load-bearing (removing either degrades or breaks the suite), correctly scoped (touches only the `jest` config block, nothing else), and does not mask a real behavioral bug — it redirects `react-native-reanimated` to its standard mock entry point for RNTL component tests, which is the documented/recommended pattern, not a workaround that hides broken app logic.
