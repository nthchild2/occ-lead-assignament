## Existing patterns to follow

- **Themed screen skeleton** → follow the placeholder itself, `app/app/(auth)/login.tsx:10-30`: `<View style={{ flex: 1, ..., backgroundColor: theme.colors.bg, padding: theme.spacing[6] }}>`, text via `theme.type.*` + `theme.colors.fg`. Replace the body, keep the wrapper convention.
- **Controlled form inputs** → `Input` is a thin wrapper over RN's `TextInput` (`app/core/components/Input.tsx:6,39-65`); use plain `useState` for `email`/`password`/`error` in the screen (no new store — spec's own resolution at `00-spec.md:36` confirms this and matches `app/store/theme.store.ts`'s "one store per domain" rule in `docs/MAP.md:44-46`, i.e. auth already has its store, login-screen doesn't need one).
- **Submit button with loading/disabled** → `Button.tsx:103-107` already disables + shows `ActivityIndicator` when `loading` is true; the screen only needs to pass `loading={submitting}` — no need to also compute `disabled` for the in-flight case.
- **Client-side pre-submit validation** → reuse `LoginRequestSchema.safeParse({ email, password })` (`packages/shared/src/schemas/auth.schema.ts:8-11`). No screen/component in this codebase does client-side Zod validation yet (no precedent found via `grep -r "safeParse" app/app app/core`), but `safeParse` is the idiomatic Zod API for this exact case: `const result = LoginRequestSchema.safeParse({ email, password }); if (!result.success) { const first = result.error.issues[0]; setError(first.message) }`. Reusing the schema avoids duplicating the email-format/non-empty-password rules that already live in `@occ/shared` (`docs/MAP.md:48-50`: "no hand-written duplicate interfaces" — the same single-source-of-truth spirit applies to validation rules, not just types). Recommendation: **reuse `LoginRequestSchema.safeParse`**, not a hand-rolled check — Zod's default messages ("Invalid email", "String must contain at least 1 character(s)") are presentable as inline `error` text on `Input`, satisfying R2 without a parallel validation implementation to keep in sync.
- **Reading `ApiError.message` on failure** → `app/core/services/api.ts:28-36` (`ApiError extends Error`, has `.code` + inherited `.message`); catch pattern precedent in `app/store/auth.store.test.ts:46-56` (`rejects.toMatchObject({ name: 'ApiError', code: ... })`). Screen's `catch (err)` should narrow with `err instanceof Error` (strict TS) before reading `.message`.
- **No manual navigation after login** → `app/app/(auth)/_layout.tsx:9-17` already redirects once `useAuthStore((s) => s.token)` becomes truthy after `login()` resolves and calls `set({ token, user })` (`app/store/auth.store.ts:23-26`). The screen's `onPress` handler should simply `await login(...)` and let the layout's reactive redirect fire — no `router.replace`/`push` call.
- **Screen-level RNTL test (optional but A4-consistent)** → no existing screen test file exists yet (all other route files, e.g. `app/app/(protected)/(tabs)/index.tsx:1-31`, are still untouched placeholders), so there is no direct screen-test precedent to copy. Closest precedents: `app/core/services/api.test.ts` and `app/store/auth.store.test.ts:1-34` for the "mock `global.fetch`, not msw" convention (`docs/MAP.md:61`), and `docs/A4:506-514` for RNTL's role/label query convention. A test would mock `useAuthStore`'s `login` (e.g. `jest.spyOn`/module mock) rather than `global.fetch`, since the screen's unit of concern is "did it call `login()` with the right args and render the right state," not the network layer.

## Constraints that apply

- All styling via `useTheme()`, no inline color/size literals — `docs/MAP.md:33`, mirrored in every existing component/screen (e.g. `app/app/(auth)/login.tsx:16-25`). This is R7 verbatim.
- `complexity: ['error', 10]` — `.eslintrc.js:26`. Keep validate/submit/catch logic linear; extract a helper (e.g. a local `validate()` function) if branching grows past the limit.
- No `any`; `@typescript-eslint/no-explicit-any: 'error'` — `.eslintrc.js:13`. The Zod `safeParse` result and caught error must be typed/narrowed, not `any`.
- Accessibility: interactive elements need `accessibilityRole` + label — `docs/MAP.md:36`, `docs/A4:499-504`. Already satisfied by `Input`/`Button`'s internals (`Input.tsx:40`, `Button.tsx:104`) as long as `label`/`Input`'s `label` prop are supplied — the screen must pass real `label` text, not leave it empty.
- One component per file / export from barrel — `docs/MAP.md:34`. Not directly applicable (the screen is a route file, not a `core/components/` addition), but if the plan extracts a form subcomponent it belongs in `core/components/` with a barrel export, per this convention.
- `core/` must not import from `app/` — `.eslintrc.js:33-44`. Not at risk here (the screen imports from `core/`, the allowed direction) but the plan must not introduce a `core/` import back into `app/app/`.
- Snapshot tests are added "after the developer considers the screen done," reviewed in the same PR — `docs/A4:33,37`. If the plan adds a snapshot test, it must be added once, reviewed, not blindly regenerated later.

## What NOT to touch

- `app/store/auth.store.ts` — `login()`'s contract (throws `ApiError`, sets `token`/`user` on success) is already implemented and tested (`app/store/auth.store.test.ts`); this ticket only consumes it.
- `app/core/services/auth.service.ts` / `app/core/services/api.ts` — network layer is out of scope; the screen calls the store action only, never `fetch`/services directly (`docs/MAP.md:72`: "No `fetch`/`axios` in components or stores").
- `packages/shared/src/schemas/auth.schema.ts` — read-only reuse of `LoginRequestSchema`; no schema changes needed for this feature.
- `app/app/(protected)/**` — protected-area screens are unrelated; do not touch while implementing login.

## Risks & unknowns

- **No keyboard-avoidance precedent exists yet.** `grep` across `app/app` found zero uses of `KeyboardAvoidingView`/`ScrollView` — every other route is still a centered placeholder `<View>`. A login form with two inputs + a button will likely need `KeyboardAvoidingView` (iOS) so the keyboard doesn't cover the password field, but there's no existing pattern to cite for this. Not a blocker — flag for the plan to make an explicit, cited-in-code decision (e.g. RN's own `KeyboardAvoidingView` API) since no in-repo precedent exists to point to.
- **No client-side Zod-validation precedent exists yet** (confirmed via `grep -r "safeParse" app/`, no hits outside `api.ts`'s error-mapping use of `ApiErrorSchema.safeParse`). The recommendation above (reuse `LoginRequestSchema.safeParse`) is based on general Zod idiom, not an in-repo precedent — this is a judgment call the plan should state explicitly rather than treat as an established convention.
- **No screen-level test is in the spec ledger** (`00-spec.md`'s requirements ledger R1–R7 contains no test requirement, unlike other tickets' typical R8/R9 test rows). This is consistent with A4's model where screen tests/snapshots are added "when the developer considers the screen done," not tracked as a separate numbered requirement — so its absence from the ledger is not an oversight. The plan should still decide whether to add an RNTL interaction test in this PR (recommended, since A4 frames this PR as the one that "completes" the screen) or explicitly defer it.

## Handoff to PLAN

- Replace `app/app/(auth)/login.tsx` in place; keep its existing themed-`View` wrapper convention, add `Input`×2 (email, password with `secureTextEntry`) + `Button` from `app/core/components` (barrel import), all local state via `useState` — no new store.
- Validate with `LoginRequestSchema.safeParse({ email, password })` from `@occ/shared` pre-submit (surface `result.error.issues[0].message` as `Input`'s `error` prop); on success call `useAuthStore.getState().login` (or the hook form) — never touch `authService`/`api.ts` directly.
- `Button`'s `loading` prop alone disables it during submission (`Button.tsx:106`); wrap the submit handler in `try { setSubmitting(true); await login(email, password) } catch (err) { setError(err instanceof Error ? err.message : 'Unknown error') } finally { setSubmitting(false) }` — do not call `router.*`, the `(auth)/_layout.tsx` redirect (already built) handles navigation once `token` is set.
- No keyboard-avoidance or client-Zod-validation precedent exists elsewhere in the repo — the plan should make both decisions explicit (recommended: `KeyboardAvoidingView` for layout, `LoginRequestSchema.safeParse` for validation) rather than treat them as "established convention."
- Decide explicitly whether to add an RNTL screen test in this PR — A4 (`docs/A4:33`) frames this as the PR that "completes" the screen, which is when screen tests/snapshots are normally added, even though the spec ledger has no dedicated test `R`-id.
