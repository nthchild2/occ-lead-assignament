## Deviations from the plan

- **Jest infrastructure fix, required to make the plan's own test runnable (orchestrator-applied after the implementer's session was interrupted).** The implementer's original commit added `"setupFiles": ["react-native-reanimated/mock"]` to `app/package.json`'s jest block to handle `Button.tsx`'s use of `react-native-reanimated` (this is the first test in the repo to `render()` a component tree that includes an animated component — all prior tests were store/hook logic, not component rendering). That specific fix was **wrong and regressive**: it broke **all 12 test suites** (not just the new one) with `WorkletsError: Native part of Worklets doesn't seem to be initialized` — `react-native-reanimated/mock` is a plain module (not a `jest.mock()` registration) that, when `require()`'d as a side-effecting `setupFiles` entry under this reanimated 4.1.7 + separate `react-native-worklets` 0.8.3 setup, still transitively touches the native worklets initializer.
  - **Root cause, diagnosed by reading the actual packages in `node_modules`:** `react-native-worklets` ships its own dedicated Jest resolver at `react-native-worklets/jest/resolver.js`, which filters out `.native.js` extensions specifically for worklets-internal modules — this is the documented fix (per the error message's own troubleshooting URL) for `WorkletsError` under Jest. Separately, `react-native-reanimated/mock` is not a `jestSetup.js`-style auto-registering mock in this version (no such file exists in the package) — it's an alternate entry point that must be wired via `moduleNameMapper` (or a per-file `jest.mock()` call) to actually redirect `import ... from 'react-native-reanimated'`, not just evaluated once via `setupFiles`.
  - **Fix applied:** removed the `setupFiles` entry; added `"resolver": "react-native-worklets/jest/resolver"` (fixes the native-init error for all 12 suites) and `"react-native-reanimated": "react-native-reanimated/mock"` under `moduleNameMapper` (properly redirects every `react-native-reanimated` import — including transitively, from `Button.tsx` — to the mock, fixing the remaining `AnimatedComponent`/`NativeEventsManager` native-view-tag error that surfaced once the worklets-init error was fixed).
  - **Verified:** `pnpm --filter './app' run test` → 12/12 suites, 63/63 tests green (11 pre-existing suites + the new `login.test.tsx`, no regressions). This fix is durable infrastructure — every future screen ticket that renders a `Button` (or any reanimated-based component) benefits from it without rediscovering the same failure.
- **One test-quality fix, also orchestrator-applied:** the invalid-email test originally asserted `screen.getByText(/email/i)`, which ambiguously matched both the "Email" field label and the "Invalid email" Zod error text, throwing "Found multiple elements." Tightened to the exact string `'Invalid email'` (Zod's actual default `.email()` message), which is unique from the label. This is a test-correctness fix, not a behavior change — `login.tsx` itself was not touched for this.
- All other plan steps (1–4) were implemented as specified. No other deviations.
