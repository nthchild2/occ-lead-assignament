## Approach

Two Express-free-service domains under `backend/src/domains/` — `applications/` and `favorites/` — mirroring the existing `jobs`/`auth` shape (`*.schema.ts` / `*.service.ts` / `*.router.ts`). Each service owns a `Map<userId, Set<jobId>>` and takes job existence via a `getJob: (id: string) => Job` function supplied by a router **factory** (`createApplicationsRouter({ getJob })`), never by importing `domains/jobs` directly. `app.ts` is the only place that imports `jobsService.getById` and wires it into both factories — this makes A1 Decision 3 compile-time true regardless of the eslint-zone gap research found (`.eslintrc.js:70-86` has no `applications|favorites → jobs` zone). Each domain exposes two Express routers: an "actions" router (mounted at `/jobs`, alongside the existing public `jobsRouter`) for `POST/DELETE /:id/apply` or `/:id/favorite`, and a "list" router (mounted at `/applications` / `/favorites`) for `GET /`. All four routers require `authMiddleware` + the `if (!req.user)` guard, copied verbatim from `auth.router.ts:26-39`.

Rejected alternative: a single combined `activity` domain (applications + favorites in one). Rejected because the spec's resolved ambiguity (`00-spec.md:39`) and research both call for two domains mirroring existing sibling domains 1:1, keeping each service small and each error taxonomy (`ALREADY_APPLIED` vs `ALREADY_FAVORITED`) cleanly separated; a combined domain would also blur the R9 injection boundary into one bigger surface for no benefit.

## Risks & rollback

- **Router-mount ordering at `/jobs`.** Two authed routers and one public router share the `/jobs` prefix. Risk: a careless route pattern (e.g. `/:id` catch-all) could shadow `/:id/apply`. Mitigation: the new routers only declare `/:id/apply` and `/:id/favorite` (never a bare `/:id` or `/`), so there is no path overlap with `jobsRouter`'s `GET /` and `GET /:id` regardless of mount order. Rollback: revert `app.ts` to the single-line `app.use('/jobs', jobsRouter)` plus the two commented placeholders (git revert of change #7) — the two new domain directories stay inert (unimported) with no effect on the app if verify fails.
- **In-memory state leaking between tests.** Both services use module-level `Map`s with no reset hook; sequential tests in one file share state by design (the dedup/404-after-delete assertions rely on this), but running the two test files in parallel jest workers is safe since they import separate service module instances per file (Node module cache is per worker) — no cross-file leakage. Risk is limited to within-file ordering; tests are written as an explicit sequential narrative (apply → dupe → cancel → re-cancel) to make this an asset, not a bug.
- **Injection boundary regression.** If a future edit adds `import * as jobsService from '../jobs/jobs.service'` directly inside `applications.service.ts` or `favorites.service.ts`, eslint will NOT catch it (confirmed gap, research "Constraints"). Mitigation for this plan: keep the `getJob` parameter as the _only_ way either service touches job data — a manual code-review check (grep for `domains/jobs` outside `app.ts`) is added as a verify-phase spot-check note rather than relying on lint. Not a rollback item since no such import is introduced by this plan.
- **Duplicate id collision across the two `Map`s.** Because applications and favorites are separate services with separate state, a job appearing in both is expected and not an error; no shared-key risk between domains.
- **If verify fails on any single router/service file**, that file is regenerated in place (create-type changes are idempotent — no destructive edits to existing files except change #7's four-line addition to `app.ts`, which is a small, easily-diffed edit to revert).
