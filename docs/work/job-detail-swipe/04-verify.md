## Worklet-boundary / UI-thread correctness (independent code-reading, not mock-trusting)

The implementer explicitly flagged that the reanimated jest mock (`react-native-reanimated/mock`, wired via `jest.config.js`'s `moduleNameMapper: { "^react-native-reanimated$": "react-native-reanimated/mock" }`, confirmed in `app/package.json`) makes `runOnJS` an identity function, so the test suite cannot prove the real UI-thread/JS-thread boundary compiles/behaves correctly. This was investigated independently:

- **Babel plugin position** — `app/babel.config.js` has exactly one plugin, `'react-native-reanimated/plugin'`, so it is trivially "last" (there is nothing after it to be out of order with). Confirmed present.
- **Worklet marking** — `grep -n "'worklet'"` across `JobDetail.tsx` returns exactly one hit, on the `Gesture.Pan().onEnd((event) => { 'worklet'; ... })` callback (line 224). This is the only code that needs to run on the UI thread (it reads `event.translationX`/`event.velocityX` — gesture-handler event data only available there — and mutates `translateX.value`, a shared value).
- **The JS-thread hop is correct** — `runOnJS(handleSwipeEnd)(direction)` (line 231) is called from inside the worklet. `handleSwipeEnd` (lines 197-214) is a **plain, non-workleted** closure defined in `useJobSwipe`'s function body (JS thread, created fresh each render) that calls `useJobsStore.getState().setActiveJob(...)` (a Zustand `set()` — must run on JS thread; Zustand has no worklet-safe API) and, transitively via `maybePrefetchNextPage`, `InteractionManager.runAfterInteractions(...)` (a core RN JS-thread-only API — this would not exist/work as a worklet). `runOnJS` is exactly the correct, and only, mechanism to make this hop. The single argument crossing the boundary, `direction: 1 | -1`, is a plain serializable primitive (not a closure over UI-thread-only state), which is a hard requirement for `runOnJS` arguments — this is satisfied.
- **No incorrect direct calls from worklet context** — confirmed nothing inside the `'worklet'`-marked callback calls `setActiveJob`, `InteractionManager`, or any other JS-thread-only API directly; the only side effect inside the worklet itself is `translateX.value = 0` (correct — shared-value mutation is UI-thread-safe/required) and the early `return` on failed threshold (correct — no JS-thread call needed for that branch).
- **Type-safety consequence noted by implementer** (`useAnimatedStyle<ViewStyle>(...)`'s explicit generic) was independently re-derived: `tsc --noEmit` passes cleanly across all three workspaces in the fresh run above, so this doesn't mask a live typing bug.
- **Conclusion: the worklet/JS-thread boundary, as written, would function correctly on a real device.** This is a code-reading conclusion, not a mock-trusting one — the mock's identity-function `runOnJS` was explicitly bypassed in this reasoning by reading the real gesture-handler/reanimated type declarations and the actual call graph.

**`useSharedValue`/`useAnimatedStyle` vs. `Button.tsx` precedent** — `useJobSwipe`'s `const translateX = useSharedValue(0)` and `useAnimatedStyle<ViewStyle>(() => ({ transform: [{ translateX: translateX.value }] }))` match `Button.tsx`'s `usePressAbsorb`'s `useSharedValue(0)` / `useAnimatedStyle(() => ({ transform: [...] , shadowOpacity: ... }))` shape exactly — same direct-from-`react-native-reanimated` import style, same "shared value in, transform-array-returning worklet callback out" convention. No divergence found.

## `.activeOffsetX`/`.failOffsetY` gesture-config reasoning

Confirmed present at `JobDetail.tsx:221-222`: `.activeOffsetX([-10, 10]).failOffsetY([-10, 10])`. Read the installed package's own doc comments (`react-native-gesture-handler/src/handlers/gestures/panGesture.ts:63-77,82-96`) to verify semantics rather than assuming:

- `activeOffsetX([-10, 10])` — the pan gesture only _activates_ once the finger crosses ±10pt horizontally; below that it stays pending.
- `failOffsetY([-10, 10])` — if the finger moves beyond ±10pt vertically **before** the gesture has activated, this pan gesture **fails outright**, releasing the touch stream to sibling recognizers.

This is the standard, textbook-correct technique for a horizontal-swipe gesture that must coexist with a vertical `ScrollView`/`BottomSheetScrollView`: any drag that reads as more vertical than horizontal causes an immediate hard yield (`fail`, not an ambiguous simultaneous-recognition race) to the scroll view's own native pan responder, which then handles the vertical scroll normally. It does **not** accidentally block `BottomSheetScrollView`'s own vertical scroll gesture — if anything it is the mechanism that actively protects it. No `simultaneousWithExternalGesture`/`blocksExternalGesture` calls are present or needed given this design. This has no automated test coverage (confirmed — correctly flagged by the implementer as outside RNTL/jest-expo's reach), but the configuration itself, read against the library's own documented behavior, is correct.
