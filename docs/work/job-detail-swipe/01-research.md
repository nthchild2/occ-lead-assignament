## Existing patterns to follow

- **UI-thread animation with reanimated** → follow `app/core/components/Button.tsx:10,44-58` (`useSharedValue` + `useAnimatedStyle` + `withTiming`, imported directly from `react-native-reanimated`). No `useAnimatedGestureHandler` exists anywhere in this codebase or in the installed reanimated 4.1.7 package — do not introduce it.
- **Declarative gesture** → the installed `react-native-gesture-handler@2.28.0` exports `Gesture.Pan()` + `GestureDetector` (`node_modules/.../react-native-gesture-handler/lib/typescript/index.d.ts:27-29`) — this is the API to use; no in-repo precedent yet (this is the first gesture ticket), so the pattern must be established here, matching the exported API confirmed above, not any legacy `PanGestureHandler` API.
- **Route-lifecycle-coupled logic placement** → follow `JobDetail.tsx:254-259`'s own doc comment: store-aware, sheet-adjacent code belongs in `app/app/(protected)/`, not `app/core/components/`, because `core/` cannot import `app/store/*`'s consumers back into `app/app/` and because this logic is inherently tied to the sheet's lifecycle (`docs/MAP.md:36`).
- **Complexity-ceiling extraction** → follow `JobDetail.tsx:112-153` (`ActionButtons` extracted out of `JobDetailContent`) and `index.tsx:71-118` (`FilterBar` extracted out of `SearchScreen`) — extract the gesture handler, prefetch-threshold check, and end-of-results indicator into separate helpers/subcomponents rather than branching inline in `JobDetailContent`.
- **Imperative store reads inside async/event handlers** → follow `useJobs.ts:26-46`'s `useJobsStore.getState()`-at-call-time idiom (never closes over stale state) — the swipe handler's prefetch-threshold check and `onDismiss`'s scroll-to-index call should read `jobs.store` the same way, not via a stale closed-over selector value.
- **Zustand store for cross-tree coordination** → follow `app/store/theme.store.ts:32-34` (`.getState()` called imperatively from outside any React component, e.g. an `Appearance` listener) as the precedent for **storing the FlashList ref itself on `jobs.store`** (a plain `RefObject<FlashListRef<Job>>` field, created once via `useRef` in `index.tsx` and registered into the store, then read via `.getState()` in `(protected)/_layout.tsx`'s `onDismiss`). This is the cleanest mechanism available in this codebase: no `React.createContext` exists anywhere in the app (would be a new, one-off pattern), and a bespoke module-level ref singleton would duplicate what the store already does for exactly this kind of cross-tree imperative handle (A3's own "Implementation note" at `docs/A3 · Navigation & Deep Linking.md:256-260` independently arrives at "a React context or a module-level variable... a Zustand slice is possible" for an analogous ref-handoff problem — recommending the Zustand-slice option here keeps the mechanism consistent with the rest of the codebase's exclusive reliance on Zustand for state/handle sharing).
- **Module-boundary Jest mocks scoped per test file** → follow `JobDetail.test.tsx:23-28` (`@gorhom/bottom-sheet` → `ScrollView` stand-in), `_layout.test.tsx:34-63` (`@gorhom/bottom-sheet`'s `BottomSheetModal` → `forwardRef` stand-in with `useImperativeHandle`), and `index.test.tsx:38-73` (`@shopify/flash-list` → stand-in forwarding `data`/`renderItem`/`onEndReached` as props). This ticket needs the same treatment for `react-native-gesture-handler` (`Gesture`/`GestureDetector` crash identically under `jest-expo` — same ESM/native-module class of failure per `docs/MAP.md`'s test guidance) scoped to whichever test file renders the swipeable `JobDetailContent`, never in the shared jest config.

## Constraints that apply

- `core/` must not import from `app/` — `.eslintrc.js:33-44` (rationale: A1 Decision 2, `docs/MAP.md:36`). The new swipe/gesture code must live under `app/app/(protected)/`, not `app/core/components/`.
- `complexity: ['error', 10]` — `.eslintrc.js:26`. Extract the gesture handler, threshold check, and end-of-results branch into named helpers.
- No `any` — `.eslintrc.js:13`. `Gesture.Pan()`'s event payload types and `FlashListRef<Job>` are both fully typed by the installed packages; no need for `any` anywhere in this ticket.
- No inline styles / theme tokens only — `docs/MAP.md:33`, consistent with all of `JobDetail.tsx`'s existing inline-`style={{ ...theme... }}` usage (e.g. `JobDetail.tsx:136-138`) — any new end-of-results indicator or swipe-affordance styling must pull from `useTheme()`.
- Only `auth.store` is persisted (A2 Decision 1) — if a `FlashListRef` object is added to `jobs.store`, it must not be persisted (it isn't — `jobs.store.ts` has no `persist` middleware, confirmed at `jobs.store.ts:41`) and, being a non-serializable ref object, must never be included in any future `persist`/`partialize` config for this store.
- No `fetch`/`axios` in stores — `docs/MAP.md:47`; the store continues to hold no network logic. `fetchNextPage()` stays owned by `useJobs.ts`; the swipe handler only calls it (already confirmed idempotent/guarded, see Relevant files above).
- `InteractionManager.runAfterInteractions` per A5 Decision 3 (`docs/A5 · Performance.md:155-171`) must wrap the prefetch call so it doesn't compete with the swipe animation's UI-thread work — `InteractionManager` is a core React Native export (`require('./Libraries/Interaction/InteractionManager').default`, confirmed present in the installed `react-native@0.81.0` package's `index.js:226-227`), no new dependency needed.

## Risks & unknowns

- **R3's premise needs correction, not blocking, but the plan must not copy it verbatim.** The spec's R3 source column says "R9 fallback sets index -1 implicitly" — this is **not what the code does**. `JobDetail.tsx`'s `useActiveJob` fallback (R9 from `job-detail-sheet`) never calls `setActiveJob`/touches `activeJobIndex` at all; it resolves `job` purely in local component state, leaving `activeJobIndex` at whatever it already was (`null` in the only real entry path today, since `setActiveJob` is called exactly once in the app, from `index.tsx:241`, always with a valid index). There is currently no code path that ever sets `activeJobIndex` to literally `-1` — that sentinel is A3's design for the not-yet-built deep-link flow (`docs/A3 · Navigation & Deep Linking.md:166,171`). **Recommendation for the plan:** treat "swipe disabled" as triggering whenever `activeJobIndex === null` OR the job at that index in `jobs.store.jobs` doesn't match `activeJobId` (covers today's fallback-fetch case) OR (if the plan wants to future-proof for the eventual deep-link ticket) `activeJobIndex === -1`. This is a genuine spec/implementation gap worth a one-line note in the plan, not an escalation — the fix is mechanical (broaden the disable-condition, comment why) and doesn't block RESEARCH → PLAN.
- **A2 Decision 4's pseudocode duplicates a guard `useJobs.fetchNextPage()` already performs internally.** The doc's `onSwipe` example inlines `if (nearEnd && pagination.hasNext && !isLoading) fetchNextPage()`, but `fetchNextPage()` (`useJobs.ts:42-46`) already returns early on the same two conditions. The plan should decide whether to keep a redundant outer check (arguably clearer intent, avoids an unnecessary `InteractionManager.runAfterInteractions` scheduling when nothing will happen) or drop it and rely solely on the hook's guard (less code, same behavior) — not a blocker, just worth the plan being explicit about which it picks, since A2's illustrative snippet and the real hook aren't identical.
- **No existing FlashList ref, no existing cross-tree ref-sharing mechanism in the codebase.** Confirmed via grep: zero `React.createContext` usage, zero module-level ref singletons. The Zustand-store-held-ref approach recommended above (new field on `jobs.store`, e.g. `flashListRef: RefObject<FlashListRef<Job>> | null` + a `setFlashListRef`-style registration action, or simply a plain mutable module-level object exported from `jobs.store.ts` itself since Zustand stores are already singletons) is a genuinely new pattern for this codebase (no existing store holds a ref-typed field) — flagging so the plan makes this an explicit, well-commented addition rather than treating it as "just another store field," since a `RefObject` violates the store's otherwise-plain-serializable-data convention (every other field is plain data). A2's own "Implementation note" section (`docs/A3 · Navigation & Deep Linking.md:256-260`) independently flags this exact class of problem for the (separate, out-of-scope) quit-state `jobId` handoff and explicitly considers "a Zustand slice... adds persistence risk" — same caution applies here: the plan must ensure this ref field is excluded from any future `persist` config (moot today since `jobs.store` has none, but worth a comment).
- **`@shopify/flash-list` installed version (2.3.2) exceeds the `^2.2.0` declared range's documented API in places** — the ref type is named `FlashListRef<T>`, not `FlashList<Job>` (that's the component type, not the ref type). Using `useRef<FlashList<Job>>(null)` would likely still typecheck loosely but `useRef<FlashListRef<Job>>(null)` is the precise, documented type for the imperative handle — the plan should use the latter.
- **End-of-results indicator has no existing component to reuse.** `core/components/` has no banner/toast-style primitive; the plan will need to decide on a minimal inline `Text`/`View` (following `JobDetail.tsx`'s existing inline-message pattern at lines 136-138) rather than inventing a new shared component for a single call site — consistent with this file's existing "scoped to this file" precedent (see `JobDetail.tsx:18-20`'s comment on `formatSalary`).

## Handoff to PLAN

- **Gesture API:** use `Gesture.Pan()` + `GestureDetector` (confirmed exported by the installed `react-native-gesture-handler@2.28.0`) driving `useSharedValue`/`useAnimatedStyle`/`withSpring`/`withTiming` from `react-native-reanimated@4.1.7` — mirror `Button.tsx:10,44-58`'s import/usage idiom. Do **not** use `useAnimatedGestureHandler` (absent from installed reanimated 4.x) or legacy `PanGestureHandler`. No new provider needed — `GestureHandlerRootView`/`BottomSheetModalProvider` are already mounted at `app/app/_layout.tsx:24-30`.
- **Placement:** new swipe logic belongs under `app/app/(protected)/` (route-lifecycle-coupled, store-aware) — likely a new component/hook that wraps `JobDetailContent` inside `JobDetail.tsx`, extracted per the complexity ceiling (`.eslintrc.js:26`), not under `core/components/`.
- **Cross-tree ref sharing (R7):** add a ref-holding field to `app/store/jobs.store.ts` (new, not yet present) that `index.tsx` registers its `useRef<FlashListRef<Job>>(null)` into on mount, and `(protected)/_layout.tsx`'s `onDismiss` reads via `useJobsStore.getState()` to call `.scrollToIndex({ index: activeJobIndex, animated: ... })` before/while calling `clearActiveJob()` — mirrors the store's existing `.getState()`-imperative-read idiom (`useJobs.ts:26-46`, `theme.store.ts:32-34`). Guard on `activeJobIndex !== null` (and, per the Risks note, treat `-1`/mismatched-job as the disabled case too).
- **Prefetch threshold + `InteractionManager` (R4/R5):** compare `activeJobIndex` to `jobs.length` (threshold 3) and call `fetchNextPage()` (from `useJobs()`) inside `InteractionManager.runAfterInteractions(...)` — `fetchNextPage()` already internally no-ops when `!pagination.hasNext || isLoading` (`useJobs.ts:44`), so an outer guard is optional/redundant (plan should pick one explicitly, see Risks).
- **R3's disable condition needs correcting from the spec's stated premise** — there is no code path today that sets `activeJobIndex` to `-1`; the actual "swipe unsafe" signal is `activeJobIndex === null` or a store/`activeJobId` mismatch (see Risks for the precise recommendation).
- **Testing:** scope a `jest.mock('react-native-gesture-handler', ...)` (and reuse the existing `@gorhom/bottom-sheet` / `@shopify/flash-list` mocks already in `JobDetail.test.tsx`, `_layout.test.tsx`, `index.test.tsx`) to whichever test file(s) render the swipeable content — same per-file, non-shared-config pattern used throughout this repo. `index.test.tsx`'s FlashList mock will need to grow a `forwardRef`/`useImperativeHandle` exposing `scrollToIndex` so a new test can assert it's called with the right index on dismiss.
