## Deviations from the plan

- **None.** All 9 planned changes were implemented as specified, including the spec-corrected
  R3 condition, the single-source-of-truth prefetch guard (no duplicated `hasNext`/`isLoading`
  check outside `fetchNextPage()`), and the `.activeOffsetX`/`.failOffsetY` gesture constraint
  called out as an implementer's-call mitigation in the plan's risk section (added to change #4
  to keep vertical scroll working inside the sheet).
- One test-only correction not anticipated by the plan text verbatim but within its own stated
  testing boundary: `JobDetail.test.tsx`'s `setActiveJobId` test helper needed an
  `'activeJobIndex' in overrides` check (rather than `??`) to let tests explicitly pass
  `activeJobIndex: null` — `??` cannot distinguish "not provided" from "explicitly `null`" when
  the override value is itself `null`. This is a test-helper bug found and fixed during
  implementation, not a production-code or plan deviation.

## Anything left for a follow-up

- The plan's own "Testing boundary" section already flags this, restated here for visibility:
  no test in this suite (or the codebase generally) simulates real finger-swipe touch physics,
  native gesture-recognizer activation, or the actual spring/threshold _feel_ of the animation —
  only the outcome logic reachable through the captured `onEnd` callback is exercised. Manual/
  device verification of swipe feel is out of reach of this automated suite and was not attempted.
- Deep-link / quit-state hydration (where a real `activeJobIndex === -1` sentinel might one day
  be set) remains explicitly out of scope per the spec, deferred to the `push-notifications`
  ticket — `isSwipeDisabled`'s mismatch branch already covers that case safely if/when it lands
  (`jobs[-1]` is `undefined`, so the `!==` check trips), so no follow-up work is required here.
