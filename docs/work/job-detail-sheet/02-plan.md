# Plan · job-detail-sheet

## Approach

Add one small service function (`jobs.service.getById`) to close R9's fetch gap, then build a single new presentational component (`JobDetail`) that renders the sheet's content (fields, Apply/Favorite buttons, loading/error branches), and wire the actual `BottomSheetModal` instance + `sheetRef`/`useEffect`/`onDismiss` directly into `(protected)/_layout.tsx` per A3 Decision 2's own code sketch (`<BottomSheetModal ref={sheetRef} onDismiss={onSheetDismiss}><JobDetail /></BottomSheetModal>`). `JobDetail` lives at `app/app/(protected)/JobDetail.tsx` (a non-route file inside the `app/` tree, not `core/components/`) — it is store-aware (`jobs.store`, `applications.store`, `favorites.store`, `activityStatus.ts`) and `core/` must not import from `app/` (`.eslintrc.js:33-40`), so it cannot live under `core/components/` without inverting that boundary. `JobDetail` itself takes no props (reads `activeJobId` and the stores directly) and owns R9's found-vs-fetch branching and R7/R8's try/catch, keeping `(protected)/_layout.tsx` limited to ref/effect/dismiss plumbing (mirrors how `index.tsx` keeps `SearchScreen` thin by extracting `ListContent`/`FilterBar`).

Rejected alternative: putting the sheet's content directly inline in `(protected)/_layout.tsx` (no separate component). Rejected because it would mix the auth-guard/hydration concern already in that file with unrelated job-detail rendering/business logic, breaking the "one component per file" and complexity-ceiling guardrails, and would make `JobDetail` untestable in isolation from the auth guard's `useEffect`/`Redirect` branches.

Rejected alternative: a generic toggle-button wrapper component for Apply/Favorite. Rejected per research — no existing precedent, `Button` already supports the needed `label`/`variant`/`disabled` props, and a wrapper would be premature abstraction for two call sites.

## Risks & rollback

- **`@gorhom/bottom-sheet` under Jest is unverified** (research flagged this). Change #7's test attempts a real render first; if `BottomSheetModal`/`BottomSheetModalProvider` crash under `jest-expo` (e.g. an ESM/native-module import failure, mirroring `@shopify/flash-list`'s issue), fall back to a module-boundary mock in that same test file only (not a shared jest-config change, per `docs/MAP.md:60`'s "no custom `transformIgnorePatterns`" constraint). This risk is scoped to the test file — if it proves un-mockable within budget, the test can assert the `useEffect`/`onDismiss` wiring logic directly (extracting it isn't needed; RNTL can still find the rendered `<JobDetail />` children even with a mocked `BottomSheetModal` shell). No production code is affected either way.
- **R9's fetch-by-id path adds local component state** (`loading`/`error`/`fetchedJob`) instead of a store field — if a future ticket (`job-detail-swipe`, `push-notifications`) needs this same data shared across components, that local state may need lifting into a store later; this plan intentionally keeps it local per the spec's own scope note ("this ticket only needs this for robustness"), so no rollback is anticipated, just a possible future refactor outside this ticket.
- **Rollback if verify fails**: changes are additive and ordered so each step is independently revertable — #1 (service function) has no callers until #2, so reverting #2 leaves #1 inert; #4 (layout wiring) is the only edit to an existing file used elsewhere (the auth guard) — if it breaks the guard's existing behavior, revert #4 alone (restore `return <Slot />`) without touching #1–#3, #5–#7. Each test file (#5–#7) is standalone and can be deleted/reverted without affecting production code.
- **Complexity ceiling**: `JobDetail.tsx`'s combined found-vs-fetch branching (R9) plus Apply/Favorite try/catch (R7/R8) risks exceeding complexity 10 in a single function — plan requires extracting the fetch-resolution logic and the Apply/Favorite handlers as separate named functions within the file (mirroring `index.tsx`'s `ListContent`/`FilterBar` extraction), not one monolithic component body.
