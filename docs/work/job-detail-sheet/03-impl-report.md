## Deviations from the plan

- **Change #6/#7 (test files): added a `jest.mock('@gorhom/bottom-sheet', …)` in _both_ `JobDetail.test.tsx` and `_layout.test.tsx`, plus an additional `jest.mock('expo-router', …)` in `_layout.test.tsx`.** The plan's Risks section anticipated this only for change #7 ("attempts a real render first; if it crashes… fall back to a module-boundary mock in that same test file only"). In practice, `JobDetail.tsx` itself imports `BottomSheetScrollView` directly, so its own test (`JobDetail.test.tsx`) hits the identical `@gorhom/bottom-sheet` → `react-native-gesture-handler` → `ReactNativeRenderer-dev.js` import-time crash under `jest-expo`, confirmed by a real-render attempt before adding the mock. Separately, `_layout.test.tsx` is the first test in the repo to render a `_layout.tsx` file (not just a screen component); `expo-router`'s real `<Slot />` calls `useRouteNode`, which uses React 19's `use()` — unavailable under this project's React 18 `jest-expo`/`react-test-renderer` combo — so `<Slot />`/`<Redirect />` were also mocked, narrowly, to just those two exports. Both mocks are scoped to their respective test files only, never the shared jest config, consistent with the plan's stated fallback rule and `docs/MAP.md`'s "no custom `transformIgnorePatterns`" constraint.
- All other plan steps (#1–#5) were implemented exactly as specified. No production-code deviations.

## Anything left for a follow-up

- Native swipe-down/backdrop-tap dismissal of the real `BottomSheetModal` is not exercised by any test (only the programmatic `onDismiss` callback is) — consistent with the plan's own Risk note and the `job-search-screen` precedent for `FlashList`'s `onEndReached`; genuinely testing native gesture-driven dismissal would require heavy `@gorhom/bottom-sheet` internals mocking, out of scope here.
- R9's fetch-by-id state is local to `JobDetail` (`useState`, not a store field), exactly as the plan intended ("this ticket only needs this for robustness"); if `job-detail-swipe` or `push-notifications` need this data shared across components later, lifting it into a store is a decision for that ticket, not this one.
- Swipe-between-jobs, transparent prefetch, end-of-results indicator, and deep-link entry/quit-state hydration remain explicitly out of scope per the spec's "Explicitly out of scope" section — not touched.
