## Existing patterns to follow

- **Owning a `BottomSheetModal` instance in a layout** → follow `docs/A3 · Navigation & Deep Linking.md:117-142`'s code sketch almost verbatim: a `sheetRef = useRef<BottomSheetModal>(null)` in `(protected)/_layout.tsx`, a `useEffect` on `activeJobId` calling `.present()`, and rendering `<><Slot /><BottomSheetModal ref={sheetRef} onDismiss={onSheetDismiss}>...</BottomSheetModal></>`. This ticket does not need the `flashListRef.current?.scrollToIndex` line (that's `job-detail-swipe`'s concern per the ledger's "Explicitly out of scope").
- **Extracting the sheet content into its own component** → the spec's resolved open question favors a dedicated `JobDetailSheet` component rendered by `(protected)/_layout.tsx`, for readability — no existing component precedent for this split since it's the first detail screen, but it mirrors the general "one component per file, keep screens thin" pattern seen in `app/app/(protected)/(tabs)/index.tsx:120-171`'s extracted `ListContent`/`FilterBar` helpers.
- **Formatting a `Job` field for display** → follow `app/core/components/JobCard.tsx:19-25` (`formatSalary` via `Intl.NumberFormat('es-MX', { style: 'currency', currency: 'MXN', maximumFractionDigits: 0 })`) and its null-salary guard (`JobCard.tsx:56-66`) and tag-badge rendering (`JobCard.tsx:67-73`, using `Badge` from `app/core/components/Badge.tsx:14`). No existing `publishedAt` formatter — will need a small new one-off (no date library installed, see Risks).
- **Async orchestration outside the store** → follow `app/core/hooks/useJobs.ts:13-16,25-54`'s `messageFor(error)` + `ApiError` narrowing pattern for any new fetch-by-id logic (R9).
- **Toggling a `Button`'s visual state based on domain state** → no existing toggle-button precedent; compose it inline by branching `label`/`variant`/`disabled` before passing to `Button` (`app/core/components/Button.tsx:18-26`), same spirit as `ListContent`'s conditional branches in `index.tsx:137-155`.
- **Adding a single-job fetch service function** → mirror `app/core/services/jobs.service.ts:23-25`'s `list()` shape: `export function getById(id: string): Promise<JobDetailResponse> { return get(`/jobs/${id}`, JobDetailResponseSchema, false) }` (unauthenticated, matching the router having no `authMiddleware` per `backend/src/domains/jobs/jobs.router.ts:25-27`). `JobDetailResponseSchema`/`JobDetailResponse` already exist in `@occ/shared` (`packages/shared/src/schemas/job.schema.ts:42-44,50`) — no shared-package change needed.
- **Mocking a third-party UI library under Jest** → follow `app/app/(protected)/(tabs)/index.test.tsx:38-73`'s `jest.mock('@shopify/flash-list', () => {...})` pattern (module-factory using `jest.requireActual` for `react`/`react-native`) if `@gorhom/bottom-sheet`'s real implementation doesn't render under `jest-expo` (needs a quick spike in PLAN/BUILD — no existing bottom-sheet test in the repo to confirm either way yet).
- **Mocking a Zustand store module in a screen/component test** → follow `app/app/(auth)/login.test.tsx:11-21`'s `jest.mock('../../store/x.store', () => ({ useXStore: { getState: jest.fn() } }))` shape; `index.test.tsx:15-17,99-106` shows the richer variant needed when a component uses both the hook-selector form (`useJobsStore((s) => s.field)`) and `getState()` imperatively — the sheet will need this richer variant for `jobs.store`, `applications.store`, and `favorites.store` simultaneously.

## Constraints that apply

- `core/` must not import from `app/` — `.eslintrc.js:33-40` (`import/no-restricted-paths`). The new `jobs.service.getById` and any `core/lib` helpers must stay import-clean; only the `(protected)/_layout.tsx` / `JobDetailSheet` component (in `app/app/`) may import stores directly.
- No `any` — `.eslintrc.js:13`. All `Job` typing flows from `@occ/shared`'s `Job` type; no hand-rolled interface for job-detail props.
- Cyclomatic complexity ≤ 10 — `.eslintrc.js:22` (`complexity: ['error', 10]`). The sheet's Apply/Favorite conditional-state logic and R9's found-vs-fetch branching should extract helpers if they start nesting deeply, mirroring `index.tsx`'s `ListContent`/`FilterBar` extraction (`(tabs)/index.tsx:120-171`).
- No inline styles/literals — all styling via `useTheme()` tokens (`docs/MAP.md:33`, `app/core/hooks/useTheme.ts:5`).
- Interactive elements need `accessibilityRole` + label — `docs/MAP.md:36` (A4 Decision 8). Apply/Favorite buttons already get this for free via `Button`'s `accessibilityRole="button"` (`Button.tsx:104`); the sheet's own scroll/close affordances should follow the same rule if any custom pressables are added.
- One component per file, exported via `app/core/components/index.ts` — `docs/MAP.md:34`, `app/core/components/index.ts:1-10`. If a new reusable primitive is extracted (unlikely — Button/Badge cover it), it goes there; the `JobDetailSheet` itself is route/store-aware so it belongs under `app/app/`, not `core/components/`.
- No `fetch`/`axios` in components or stores — `docs/MAP.md:72`. `getById` belongs in `app/core/services/jobs.service.ts`, called from the sheet component or a small hook, not inlined.
- Only `auth.store` is persisted; other stores reset on launch — `docs/MAP.md:45` (A2 Decision 1). No change needed here, just don't add persistence to `jobs.store`.
- App tests are co-located (`*.test.tsx` next to the unit) — `docs/MAP.md:59`. New test file: `app/app/(protected)/_layout.test.tsx` and/or a co-located test next to wherever `JobDetailSheet` lives.
- Do not add a custom `transformIgnorePatterns` to the app jest config — `docs/MAP.md:60`. If `@gorhom/bottom-sheet` fails to transform under Jest, mock it at the module boundary instead (see pattern above), do not touch `app/package.json`'s `jest` config transform settings.

- Backend `backend/src/domains/jobs/*`, `backend/src/domains/applications/*`, `backend/src/domains/favorites/*` — all needed endpoints (`GET /jobs/:id`, apply/cancel, favorite/unfavorite, 409 codes) already exist and are out of scope per the spec ("Backend changes" explicitly excluded).

- Deep-link (`occ://vacante/:id`) URL-param handling (`useLocalSearchParams`, `router.setParams`) — explicitly `push-notifications`' scope (A3 Decision 3). R9 only covers "job not in `jobs.store.jobs` by id" as a defensive fallback, not the full deep-link entry flow.

## Risks & unknowns

- **`@gorhom/bottom-sheet` under Jest is unverified.** No existing test in the repo exercises `BottomSheetModal`/`BottomSheetModalProvider`. It may render fine under `jest-expo` (it's a real RN component tree, not ESM-only like `@shopify/flash-list`), or it may need the same module-boundary mock treatment. This should be resolved empirically in BUILD/PLAN with a quick spike rather than assumed — not a blocking ambiguity, just a note that the test strategy may need adjustment once the real render is attempted.
- **No `publishedAt` formatting precedent exists anywhere in the app.** `JobCard.tsx` doesn't render it, and no date-formatting library (`date-fns`, `dayjs`) is installed (`app/package.json` dependencies checked — none present). The plan should decide on a minimal inline formatter (e.g., raw ISO slice, or a small `Intl.DateTimeFormat('es-MX', {...})` call) rather than adding a new dependency for one field.
- **R9's scope is intentionally narrow** — the spec and A3 both point to a `jobsService.getById(id)` shape, but A3's own code sample (`A3:159-169`) is for the _full_ deep-link flow (`useLocalSearchParams`, `setActiveJob(job.id, -1)`), which is explicitly out of scope here. The plan must scope R9 to: "if `activeJobId` is set but no job with that id exists in `jobs.store.jobs`, call the new `getById` and hold the result locally (or via a minimal local state/hook) to render the sheet" — without wiring URL params or building a new store/hook ticket, per the spec's own resolution note.
- **Button toggle-state composition is unresolved by any existing component** — confirmed no wrapper exists; the plan should keep this simple (conditional `label`/`variant`/`disabled`/`onPress` passed to the existing `Button`), not invent a new abstraction, consistent with the spec's own resolution note on this open question.
