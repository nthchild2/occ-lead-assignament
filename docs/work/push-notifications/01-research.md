## Existing patterns to follow

- **Store-driven sheet-open machinery** → follow `app/app/(protected)/(tabs)/index.tsx:254-256`'s `useJobsStore.getState().setActiveJob(id, index)` call. Every entry point into the Job Detail sheet (card tap, applied-row tap, favorite-row tap, and now notification tap / deep link) funnels through this one store action — do not introduce a parallel mechanism.
- **R9 fallback-fetch for "job not in list"** → already built in `app/app/(protected)/JobDetail.tsx:50-103` (`useActiveJob`). A notification/deep-link job will almost always hit this fallback branch (its id usually isn't in the currently-loaded `jobs.store.jobs` array) — no changes needed here, just confirm the id set via `setActiveJob` is correct.
- **Hydration-gated rendering** → follow `app/app/(protected)/_layout.tsx:70-81,97-110`'s `hydration` state machine (`idle → checking → ready`). Any quit-state navigation logic must wait for `hydration === 'ready'` the same way the sheet-present effect (line 87-91) implicitly does (it can't run meaningfully until `<Slot/>` renders, which requires `hydration === 'ready'`).
- **Native-module jest mocking, module-boundary style** → follow `app/app/(protected)/_layout.test.tsx:34-57` (`@gorhom/bottom-sheet`) and `:73-97` (`react-native-gesture-handler`) for the _hand-rolled_ stand-in pattern when a package needs one. For `@notifee/react-native` specifically, prefer its **own shipped mock** (`app/node_modules/@notifee/react-native/jest-mock.js`) via `jest.mock('@notifee/react-native', () => require('@notifee/react-native/jest-mock'))`, mirroring how `@react-native-async-storage/async-storage`'s own mock is wired via `moduleNameMapper` in `app/package.json:48`.
- **Expo Router imperative param-setting from a non-route context** → `router.setParams({ jobId })` (`app/node_modules/expo-router/build/imperative-api.d.ts:70`) is the mechanism to use inside a notifee event handler (which is not a component and can't call `useLocalSearchParams` itself) to hand off a jobId into the layout tree.
- **Zustand store, one per domain, not persisted** → `app/store/jobs.store.ts:51-84` — if a plan step needs new state (e.g. a "pending jobId" concept), A3's own note (`docs/A3:256-258`) explicitly warns against a Zustand slice for the quit-state handoff (persistence risk: must never survive an app restart on its own) — use a module-level variable or ref instead, per A2 Decision 1's non-persistence default combined with A3's explicit caution.

## Constraints that apply

- `core/` must not import from `app/` — `.eslintrc.js:33-44`. Not directly relevant here since all new notifee/deep-link logic is route/layout-level (`app/app/...`), not `core/`, but if a `core/services/notifications.service.ts`-style wrapper is introduced for `createChannel`/`displayNotification`, it must not import from `app/app/` or any store — it can be called _from_ a store/layout, not the reverse.
- Cyclomatic complexity ≤ 10 per function — `.eslintrc.js:26`. `(protected)/_layout.tsx` already combines guard + hydration + sheet-present effect (4 concerns); adding notifee foreground-event registration + quit-state jobId consumption will likely push it over the limit — extract helper functions (mirrors existing `handleSheetDismiss` extraction at `app/app/(protected)/_layout.tsx:43-53`).
- No `console.*` in app code — `.eslintrc.js:18`. Any dev-only "trigger a demo notification" entry point (R2) must not use `console.log` for debugging; use inline UI feedback if needed.
- No `any` — `.eslintrc.js:13`. Notifee's own types are fully typed (`Event`, `EventType`, `InitialNotification`, `NotificationAndroidChannel`, etc.) — no `any` casts should be needed for its API.
- Only `auth.store` is persisted (AsyncStorage); all other stores reset on launch — `docs/A2 · State & Data Strategy.md:40` (A2 Decision 1). Any new state this ticket introduces (e.g. a pending-jobId holder) must **not** be added to a persisted store — reinforced explicitly for this exact scenario by `docs/A3 · Navigation & Deep Linking.md:256-258`.
- `@notifee/react-native` 9.1.8 ships **no Expo config plugin** (confirmed by absence of `app.plugin.js`/`plugin/` in the installed package) — the existing `app.json:23` plugin entry `"@notifee/react-native"` will be a no-op (or possibly a build warning) under `expo prebuild`/EAS Build since there's nothing for Expo's config-plugin resolver to execute for that package name. This is a pre-existing state from `app-nav-shell`, not something this ticket is required to fix, but the plan should note it rather than assume the plugin entry is doing real work.

## Risks & unknowns

- **A3's documented `-1` sentinel for `activeJobIndex` conflicts with the actual built swipe/dismiss code.** A3 (`docs/A3:166`) says pass `-1` for a deep-link-arrived job; the real `isSwipeDisabled` (`JobDetail.tsx:122-129`) checks `activeJobIndex === null` (not `-1`) to fully disable swipe, and `handleSheetDismiss` (`_layout.tsx:43-53`) only skips `scrollToIndex` when `activeJobIndex === null` — so `-1` would pass that guard and call `flashListRef.current.scrollToIndex({ index: -1 })`, which is invalid input to FlashList. **Not a blocking ambiguity** — resolved here: the plan should pass `-1` as the index (consistent with A3's intent and with `isSwipeDisabled`'s second condition, which also naturally disables swipe since `jobs[-1]` is `undefined`), AND additionally guard `handleSheetDismiss` (or the dismiss path generally) to only call `scrollToIndex` when `activeJobIndex !== null && activeJobIndex >= 0`. This is a small, mechanical fix to existing code the plan must include as an explicit change, not a new design question.
- **No dynamic route file matches `occ://vacante/:id` today, and Expo Router requires an actual route (or `+not-found`) to resolve an incoming URL.** Concrete recommendation (resolved, not escalated): add `app/app/(protected)/vacante/[id].tsx` — a route file that renders nothing itself and, on mount, reads `id` via `useLocalSearchParams<{ id: string }>()`, calls `useJobsStore.getState().setActiveJob(id, -1)`, and immediately `router.replace('/(protected)/(tabs)')` (or `<Redirect .../>`) so the URL bar/history collapses back to the tab screen with the sheet open over it — mirroring A3's intent (sheet opens over whatever tab is active) without literally using `?jobId=` query params on the tabs route (A3's snippet), since a **dedicated dynamic segment route** is the mechanism Expo Router actually auto-registers as a deep-link target from file structure (confirmed via `app/node_modules/expo-router/build/matchers.d.ts:13` — group folders are stripped from the path, so `(protected)/vacante/[id].tsx` → URL path `/vacante/:id`, exactly matching `occ://vacante/:id`). This also naturally covers R6 (a bare `occ://vacante/:id` tap from outside the app, no notification involved) with no notifee dependency at all. The route lives under `(protected)` so the auth guard applies to it (an unauthenticated deep-link tap redirects to login first, same as any other protected route).
- **The quit-state jobId handoff does not exist yet anywhere in the codebase** (confirmed: zero `notifee`/`Linking` references in `app/app`, `app/store`, `app/core`). A3's own note (`docs/A3:256-258`) flags this needs deliberate design. Concrete recommendation (resolved, not escalated): a module-level mutable variable (not a ref, since there's no single component instance to own a `useRef` across `app/_layout.tsx` and `(protected)/_layout.tsx` — they're siblings in the tree, not parent/child in a way that shares a ref) in a small new file, e.g. `app/core/lib/pendingNotification.ts`, exporting `setPendingJobId(id: string | null)` / `consumePendingJobId(): string | null` (read-and-clear, so it's only ever applied once). `app/_layout.tsx` calls `notifee.getInitialNotification()` in a `useEffect` on mount and calls `setPendingJobId` if a `jobId` is present in the payload's `data`. `(protected)/_layout.tsx`'s existing hydration effect (`_layout.tsx:70-81`) calls `consumePendingJobId()` only after `hydration` flips to `'ready'`, and if non-null, calls `setActiveJob(id, -1)`. This keeps the value out of any persisted store (per A2 Decision 1 and A3's explicit caution) while still surviving the brief window between the two layouts mounting.
- **`onBackgroundEvent` has no unsubscribe / must be registered exactly once at module scope**, per its real implementation (`NotifeeApiModule.ts:480-485`: sets a single module-level `backgroundEventHandler`, warns via `console.warn` if none is set when the native side fires — that `console.warn` is inside `node_modules`, not app code, so it doesn't trip `no-console`). This means `onBackgroundEvent` should be registered once at module load (e.g. top-level in `app/_layout.tsx` or a small notifications-setup module), not inside a component `useEffect` that could re-run — the plan should pick one registration site and document why, to avoid duplicate/dropped handlers across re-renders.
- **The existing `app.json` notifee plugin entry is inert** (no config plugin ships with 9.1.8). This is a carry-over gap from `app-nav-shell`, not created by this ticket, but the plan should decide whether to leave it as a harmless no-op string or flag it for later removal — not fix it now, since Android channel creation happens at runtime via `notifee.createChannel()` regardless (confirmed working without any build-time plugin).
- **R2's "demo trigger" entry point has no natural hook** (no backend push source exists). Per the spec's own resolution note, keep this minimal — e.g. a small dev-only button on the search screen or a call fired shortly after successful login/hydration. This is a plan-level decision, not a research gap.
