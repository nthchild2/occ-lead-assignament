- **Date:** 2026-07-01

## Approach

Wire `@notifee/react-native` 9.1.8 (already installed) as a thin, typed service (`core/services/notifications.service.ts`) that owns channel creation and the demo trigger, keeping `core/` free of any `app/`-tree import per the eslint boundary. Route resolution reuses the app's one existing sheet-open mechanism (`jobs.store.setActiveJob`) via a new dynamic route `app/(protected)/vacante/[id].tsx`, so foreground/background/deep-link taps all converge on the same call — no parallel "open the sheet" path is introduced. Quit-state is the only case needing new cross-layout state; it gets a dedicated non-persisted module-level holder (`core/lib/pendingNotification.ts`) rather than a Zustand slice, consumed only after `(protected)/_layout.tsx`'s existing hydration gate reaches `'ready'`. The demo trigger (R2) fires once after a successful login rather than adding a dev-only button/screen — smallest footprint, no new UI surface, no styling burden, and it's a realistic proxy for "a new job notification arrives while using the app."

Rejected alternative: A3's originally-documented design (`?jobId=` query param on the tabs route, read via `useLocalSearchParams` inside `(tabs)/index.tsx`). Research confirmed Expo Router 6 auto-registers dynamic-segment route files as deep-link targets from file structure alone, and a dedicated `vacante/[id].tsx` route needs no query-param plumbing on an already-complex tabs screen, keeps the URL `occ://vacante/:id` matching literally (not `?jobId=`), and lets the auth guard on `(protected)` apply uniformly. Rejected a dev-only "send notification" button as the demo trigger: it would need its own themed UI, a reachable location, and adds a screen for a one-shot demo need that a post-login call satisfies with zero new UI.

**Explicit test boundary (mirrors `job-detail-swipe`'s "can't test real touch physics" precedent):** notifee's own `jest-mock.js` makes `createChannel`, `displayNotification`, `onForegroundEvent`, `onBackgroundEvent`, and `getInitialNotification` fully mockable as function calls — so channel-creation shape, trigger-call shape, event-handler _registration_, and the pending-jobId/deep-link/dismiss-guard logic that consumes their results are all testable under Jest. What is **not** testable in this suite: real OS-level notification delivery, the native tray UI, and an actual user tap on a system notification (foreground/background/quit) — those require a physical/emulated device and manual or E2E (e.g. Detox) verification, out of scope for this unit/integration test layer. Tests assert "given this event payload was received, this call happened" — not "the OS actually delivered and the user actually tapped."

## Risks & rollback

- **Complexity budget on `(protected)/_layout.tsx`.** Adding foreground-event registration + pending-jobId consumption on top of the existing guard/hydration/sheet-present logic risks exceeding the eslint `complexity: 10` limit. Mitigation: extract two named helpers (e.g. `handleForegroundPress(event)`, `consumePendingIfReady()`), same pattern as the file's existing `handleSheetDismiss` extraction (change #6 is scoped to do this from the start, not as an afterthought). Rollback: if verify still flags complexity, split further (e.g. a small custom hook `useNotificationDeepLink()`) rather than suppressing the rule.
- **`onBackgroundEvent` registered more than once.** Its real implementation keeps a single module-level handler with no unsubscribe; registering it inside a component effect that re-runs (e.g. on a dependency change) would silently overwrite rather than duplicate, but doing it inside a _cleanup-less_ effect that fires more than once per mount is unnecessary risk. Mitigation: register it at module scope (top of `app/_layout.tsx`, outside the component function) or in a `useEffect(() => {...}, [])` with an empty dependency array, executed exactly once per app process. Rollback: none needed at runtime (idempotent overwrite), but a test (change #14) catches accidental re-registration during development.
- **`-1` as the sentinel index interacting with FlashList/`isSwipeDisabled`.** Confirmed by research as safe against `isSwipeDisabled` (already disables swipe for any index that doesn't resolve to a match) but unsafe against the current `handleSheetDismiss`'s `scrollToIndex` call. Change #4 fixes this explicitly before change #6/#7 introduce `-1` as a real produced value — ordered so the codebase never has a window where `-1` is producible without the guard existing. Rollback: revert change #4 only requires reverting one function; it has no downstream dependents.
- **Demo trigger firing on every login is a global default, not user-toggleable.** Acceptable per spec (explicitly out of scope: "no notifications settings screen"), but if this proves noisy in manual testing, rollback is a one-line removal of the `triggerDemoJobNotification` call in `login.tsx` — it's a single call site with no other coupling.
- **`app.json` plugin entry left inert.** No action taken (by design, per research) — no rollback needed since nothing is changed. Flagged here only so a future ticket doesn't need to re-discover this.
