## Deviations from the plan

- **Implementation was interrupted by a session/connection failure before the implementer wrote this report or ran a final verification pass.** The orchestrator picked up from the committed/on-disk state and found:

  1. **The `moduleNameMapper` entry for `@notifee/react-native` was fundamentally broken, not merely untested.** The implementer's original entry pointed at the package's own documented `jest-mock.js` path — but that file is itself an ES module (`import { version } from './dist/version'` etc.) and crashes with `SyntaxError: Cannot use import statement outside a module` under this project's jest-expo transform config. This is the same class of ESM-under-Jest incompatibility hit with msw (`app-api-client`), `@shopify/flash-list` (`job-search-screen`), and `react-native-reanimated` (`login-screen`) — confirmed **repo-wide**, not file-specific, by independently reproducing the identical crash on the implementer's own `notifications.service.test.ts` before touching anything.
     - **Fix:** wrote a small hand-rolled CommonJS mock at `app/jest/notifeeMock.js` (`createChannel`/`displayNotification`/`getInitialNotification`/`onBackgroundEvent`/`onForegroundEvent` as `jest.fn()`s, plus `EventType`/`AndroidImportance` enum values copied from the real package's type declarations so any test asserting on raw values still matches production semantics) and repointed `moduleNameMapper` at it. This is a **correction to an already-present, already-broken config entry** — the same class of fix as `login-screen`'s reanimated worklets-resolver correction — not new shared-config surface area.
     - Verified: `notifications.service.test.ts` went from a hard parse-time crash to 3/3 passing with no other changes.

  2. **`app/app/_layout.test.tsx` did not exist** — the plan called for it ("new, if one doesn't exist"), but the implementer's session ended before creating it. Written by the orchestrator: 6 tests covering R1 (channel init on mount), R4 (background-event handler registration + stash/skip), and R5 (`getInitialNotification` stash/skip). Also required two additional scoped mocks beyond notifee itself — `@gorhom/bottom-sheet` (transitively crashes on `react-native-gesture-handler` under jest-expo, the same issue `(protected)/_layout.test.tsx` already solved) and `useThemeFonts` (the real `expo-font` hook throws under this test environment regardless of font-loading state, unrelated to this ticket's scope) — both scoped to this one test file, never the shared jest config.

  3. **`(protected)/_layout.test.tsx` had zero test coverage for R3 (foreground press) and R5 (pending-notification consumption)**, even though the corresponding production code (`handleForegroundPress`, `consumePendingIfReady`) was correctly implemented. The plan explicitly called for this coverage; the implementer's session ended before adding it. Added by the orchestrator: 4 new test cases (foreground `PRESS` → `setActiveJob`; non-`PRESS` → no-op; pending id consumed only once hydration is `'ready'`, not before; no pending id → no call). Required extending the file's shared `setActiveJobId` test helper to also stub `setActiveJob` on the mocked store (additive change — the 9 pre-existing test cases in this file were unaffected and still pass unmodified) and mocking `../../core/lib/pendingNotification` (previously unmocked, meaning the real read-and-clear implementation was silently exercised with no way to control its state from a test).
     - **Independently mutation-tested the trickiest of the four new assertions**: temporarily removed the `hydration !== 'ready'` early-return from `consumePendingIfReady` (so it always consumes, hydration state ignored) — the "not before hydration ready" test failed exactly as expected (2 unwanted calls instead of 0), all 16 other tests in both layout test files were unaffected. Reverted; confirmed byte-identical via `diff` against a pre-mutation backup.

  4. **A first VERIFY pass FAILED and found a real gap the orchestrator's own completion work had missed.** The plan's "Tests to add or update" section explicitly named a test case for `activeJobIndex: -1` on the `handleSheetDismiss` guard (item 3 above); the orchestrator added the `-1` guard's _sibling_ R3/R5 coverage but never actually added this specific case — the impl-report's original text claimed it was "covered by the pre-existing R7 tests," which the verifier correctly identified as misleading (those tests use `activeJobIndex: 4` and `null`, never `-1` — the one new value this ticket's guard exists to handle). The verifier independently proved the gap by removing the `>= 0` clause from the guard and confirming **zero** tests failed.
     - **Fix:** added `onDismiss skips scrollToIndex without throwing when activeJobIndex is -1 ... (R7, push-notifications)` to `(protected)/_layout.test.tsx`, mirroring the existing `null`-case test exactly. Re-ran the same mutation (removed `>= 0` from the guard) and confirmed this new test — and only this test — now fails with the exact symptom the guard prevents (`scrollToIndex({ index: -1, animated: false })` called once). Reverted; confirmed byte-identical via `diff`. Full suite re-run: 24 suites / **153** tests (152 + the new case), typecheck clean.

- **No production logic was changed beyond what the plan specified.** The bug-fix guard in `handleSheetDismiss` (plan step 4) and all notification wiring (steps 2, 3, 5, 6, 7, 8) were already correctly implemented by the time the orchestrator picked this up — verified by reading every changed source file directly, not just trusting the prior partial summary. The orchestrator's only source-adjacent change was removing one unused variable (`mockReact` in `app/_layout.test.tsx`'s gesture-handler mock) to satisfy `tsc --noEmit`.

## Anything left for a follow-up

- **`DEMO_JOB_ID = 'demo-job-1'`** in `login.tsx` is a placeholder string, not guaranteed to correspond to a real seeded job — acceptable per the spec's explicit "no real backend push source" scope, but a real deployment would want this wired to an actual job id (e.g. the newest job from a subsequent fetch) rather than a hardcoded placeholder.
- **iOS notification permission priming** — explicitly out of scope per `00-spec.md`; `notifee.requestPermission()` is not called anywhere. A real device would need this wired for the demo notification to actually display on iOS.
- This is the final ticket in the roadmap — no further follow-up tickets exist. Any remaining polish is a candidate for a future, separately-scoped chore.
