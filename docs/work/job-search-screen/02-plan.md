# Plan · job-search-screen

- **Feature id:** `job-search-screen`
- **Inputs read:** `00-spec.md`, `01-research.md`
- **Planner:** planner (agent)
- **Date:** 2026-07-01

## Approach

Build one new reusable component (`JobCard`) and one screen (`app/app/(protected)/(tabs)/index.tsx`) that composes existing, already-built primitives (`Input`, `Select`, `JobCardSkeleton`, `ErrorState`, `EmptyState`, `FlashList`) with the already-built `jobs.store`/`useJobs`/`useDebounce`. No store, hook, or shared-package edits — this ticket is pure UI wiring per spec's explicit out-of-scope list. `JobCard` is extracted to `core/components/` (not screen-local) because it is themeable, reusable (the future `job-detail-sheet` ticket may reuse it or its visual language), and MAP's "one component per file, exported from the barrel" convention applies directly — the rejected alternative (inline JSX inside the screen's `renderItem`) would fail A5's memoization requirement (you cannot cleanly `React.memo` an inline closure) and would bloat the screen's cyclomatic complexity past the ESLint ceiling of 10. All filter-to-fetch wiring (search debounce, city select, salary debounce, sort select) uses one consistent `useEffect`-on-debounced-value pattern for search and salary (both free-text/numeric entry, both benefit from not firing a request per keystroke) and direct `onChange` handlers for city/sort selects (discrete, already-debounced-by-construction — a tap is a single, deliberate event, not a keystream). A screen-level RNTL test is included (see Tests section) scoped to render/branch/interaction assertions, explicitly excluding FlashList's native scroll/pagination internals.

### Note on change #3's city option list

`be-jobs`/`JobFilters` has no "distinct cities" endpoint in scope (confirmed out of scope by both spec and research). The city `Select`'s option list is therefore necessarily a small static list defined in the screen (not derived from live data) — this is a plan-level, non-blocking decision, not scope creep, since R2 only requires "a city filter using `Select`," not a dynamically-sourced one. If IMPLEMENT finds a city list already seeded elsewhere (e.g. fixture/mock data used by `be-jobs`), it should reuse that instead of inventing new city names.

## Risks & rollback

- **`estimatedItemSize` temptation**: A5's doc still shows it in illustrative snippets; if IMPLEMENT copies it verbatim, `tsc --noEmit` fails (prop not in installed `FlashListProps<TItem>`). Rollback: remove the prop; no functional loss since v2 auto-sizes.
- **City option list is static/invented**: if a reviewer flags the city list as not matching real backend data, this is a low-cost fix — swap the static array for whatever seed data `be-jobs` actually uses, no store/hook signature changes needed.
- **`Select`'s non-optional `value: T`** could tempt an `as T` cast or a stray `!` for the city sentinel; rollback/guard: the sentinel `'__all__'` is a real, always-present string value, so no cast/non-null-assertion is needed at all — `value` is always defined by construction.
- **Complexity ceiling (10)** on the screen component: if the branch ladder (loading/error/empty/list) plus all filter handlers pushes complexity over 10, IMPLEMENT must extract a subcomponent (e.g. `<FilterBar>` or a `renderContent()` helper) rather than suppressing the ESLint rule. Rollback: extract, re-lint, no behavior change.
- **Debounce-triggered double-fetch on mount**: the mount-time `refetch()` (change 3f) and the debounced search/salary `useEffect`s (change 3a-b) must not both fire on initial render. Guard: the debounced `useEffect`s should compare against the _previous_ debounced value (or skip on first render via a ref) so mount doesn't trigger a second, redundant `refetch()` immediately after the intentional mount-time one. If this guard is missed, worst case is one harmless extra fetch, not a correctness bug — cheap to fix in review.
- **FlashList `onEndReached` test coverage gap**: acknowledged and intentional (see Tests section) — not a rollback risk, but flagged so a reviewer doesn't mistake the omission for an oversight.
