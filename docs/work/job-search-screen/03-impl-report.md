## Deviations from the plan

- **`JobCard`'s prop signature** — the plan's "Planned changes" table specified `JobCard({ job, index, onPress: (job, index) => void })`, but this ticket's explicit build instructions (which take precedence, matching the original spec's R9 acceptance criterion) specified `{ job: Job, onPress: () => void }`. Implemented the latter: `JobCard` itself is index-agnostic; the screen's `renderItem` closes over `index` and passes a zero-arg `() => onJobPress(item, index)` callback. This is a simpler, equally correct contract — no functional loss, and it decouples `JobCard` from needing to know about list position at all (arguably cleaner per single-responsibility). No R-id impact.
- **`@shopify/flash-list` module mock added to the screen test** — not anticipated by the plan's tests section (which only flagged the _scroll-triggered onEndReached_ coverage gap, not a build-time failure). At implement time, importing the real `@shopify/flash-list` package inside a Jest-run test crashed with `SyntaxError: Cannot use import statement outside a module`, because the installed v2.3.2's `dist/index.js` is published as ESM and `jest-expo`'s default `transformIgnorePatterns` only whitelists `react-native`/`@react-native*` packages (confirmed by reading `node_modules/react-native/jest-preset.js`). Per `docs/MAP.md`'s explicit instruction not to add a custom `transformIgnorePatterns` to the app's shared jest config, the fix is scoped to this one test file: a `jest.mock('@shopify/flash-list', ...)` factory (using `jest.requireActual` for `react`/`react-native`, per Jest's module-factory hoisting rules) renders `data` via `renderItem` inside a plain `View` and forwards `onEndReached` as a readable prop. This is the same "mock the module boundary" pattern MAP already prescribes for `jobs.store`/`useJobs`, applied to a third dependency the plan didn't foresee needing it. No production code changed; no shared config changed.
- Everything else: **None** — the plan's change list (JobCard creation, barrel export, screen replacement, screen test) was followed as written, including the file/store/hook boundaries (no `jobs.store`/`useJobs`/`useDebounce`/`@occ/shared` edits) and the complexity-ceiling mitigation (extracted `FilterBar`/`ListContent` subcomponents, mirroring `Button.tsx`'s helper-extraction pattern).

## Anything left for a follow-up

- The job detail bottom sheet (reading `activeJobId`/`activeJobIndex` off `jobs.store`) is explicitly out of scope per the spec (`job-detail-sheet`, a separate ticket) — not built.
- The static city option list (10 cities + "all") mirrors `backend/src/domains/jobs/jobs.seed.ts`'s current seed data but is not sourced from a live "distinct cities" endpoint (none exists in scope, confirmed by both spec and plan). If the seed data changes, this list would need a manual update — a candidate for a future ticket if a real endpoint is ever added.
- No dedicated pull-to-refresh / manual refresh affordance beyond the filter-triggered `refetch()`s and the retry button — not requested by any R-id.
