## Existing patterns to follow

- **Screen composition (form + store action + loading/error UI)** → follow `app/app/(auth)/login.tsx:25-109`: local `useState` for input values, call the store's async action via `useStore.getState().action(...)` inside a `try/catch/finally`, render inline `theme.*`-sourced styles (no `StyleSheet.create` requirement, both styles seen in the codebase).
- **Building a UI component** → follow `app/core/components/Card.tsx:14` (simple, accepts `onPress`) for `JobCard`'s outer container; consume tokens via `useTheme()` (`app/core/hooks/useTheme.ts:5`), no inline color/spacing/type literals. One component per file, export added to `app/core/components/index.ts:1-9`.
- **Memoized list item for FlashList** → `docs/A5 · Performance.md:80-93`'s `React.memo(JobCard, (prev, next) => prev.job.id === next.job.id)` pattern is still valid under v2 (memoization is a React concern, independent of FlashList's internal version) — apply it to whatever `JobCard` component this ticket creates.
- **Selector usage for filters** → `app/core/components/Select.tsx:19` doc comment names this exact component as intended for "the city filter and the five-option sort selector on Job Search" — use its `SelectOption<T>[]` shape directly for both.
- **Debounced input → filter → refetch** → `app/core/hooks/useDebounce.ts:10`; the debounced value should live in a `useEffect` that calls `setFilters({...})` then `refetch()` when it changes (no existing example in the codebase yet — this is new wiring, but both primitives are proven independently).
- **Zustand store consumption in a component** → components should call the store via a subscribing selector for read state (`useJobsStore((s) => s.jobs)`, etc.) and via `useJobs()` (a purpose-built hook, not raw store calls) for async actions — mirrors `useJobsStore`/`useJobs` being deliberately split (store = state, hook = orchestration) per `app/core/hooks/useJobs.ts:18-24`'s doc comment.
- **Test pattern if this ticket adds a screen test** → follow `app/app/(auth)/login.test.tsx:11-15`: `jest.mock('../../store/jobs.store', ...)` (and/or `../../core/hooks/useJobs`) at the module boundary, assert on rendered text/roles via RNTL, not on internal store wiring.

## Constraints that apply

- `core/` must not import from `app/` — `.eslintrc.js:33-44` (A1 boundary). `JobCard` (if placed in `core/components/`) must only depend on `@occ/shared` types and `core/hooks`/`core/theme`, never on anything under `app/app/`.
- No `any`; API types via `z.infer<>` — `.eslintrc.js:13`, satisfied automatically by importing `Job`/`JobFilters` from `@occ/shared`.
- Cyclomatic complexity ≤ 10 per function — `.eslintrc.js:26`. The screen will have several conditional render branches (loading/error/empty/list); extract subcomponents (e.g. a dedicated `<SearchScreenList>` or keep the branch as a single early-return ladder) rather than nesting.
- No inline styles outside token-sourced values — MAP "build a UI component" convention (`docs/MAP.md:33`); every color/spacing/radius/type value must come from `theme.*`.
- Accessibility: interactive elements need `accessibilityRole` + label — `docs/MAP.md:36` (A4 Decision 8). `JobCard`'s tap target needs `accessibilityRole="button"`; `Select`already provides this (`Select.tsx:49`).
- No `fetch`/`axios` in components — MAP guardrail (`docs/MAP.md:73`). All data access must go through `useJobs()`/`jobs.store`, never `jobs.service` directly from the screen.
- App tests, if added, are co-located `*.test.tsx` next to the screen (`docs/MAP.md:59`) and must mock the store/hook module boundary, not `global.fetch` or msw (`docs/MAP.md:61`).

- `app/app/(protected)/(tabs)/_layout.tsx` — tab wiring/title already correct for the `index` route; no change needed.
- Backend (`backend/src/domains/jobs/**`) — `be-jobs` already implements `GET /jobs`; explicitly out of scope per spec.
- The bottom sheet / job detail UI — `job-detail-sheet`/`job-detail-swipe` are separate, later tickets. This screen only calls `setActiveJob(id, index)` on tap; it must not render or navigate to any detail UI itself.
- `packages/shared/src/schemas/job.schema.ts` — schema is already complete for this screen's needs (all fields R1-R10 require already exist); no shared-package change anticipated.

## Risks & unknowns

- **A5's `estimatedItemSize={88}` optimization is stale against the installed FlashList v2.3.2.** The prop does not exist in the installed `FlashListProps<TItem>` type (`node_modules/.pnpm/@shopify+flash-list@2.3.2.../dist/FlashListProps.d.ts:31-357` has no such field) and the package's own README states v2 requires no size estimates and auto-sizes items (`README.md:20,37`). **Resolution (non-blocking):** the plan should omit `estimatedItemSize` entirely — passing it is not a TS error (excess-prop checks don't apply to spread `{...rest}` scenarios, but a literal JSX prop not in the interface **will** fail `tsc --noEmit` under strict mode) and is unnecessary under v2. The other four A5 optimizations (memoized `JobCard`, `keyExtractor={item => item.id}`, `getItemType` by salary presence, `drawDistance={250}`) all match the real v2.3.2 API and should be applied as documented.
- **`JobCard` does not exist anywhere in the codebase yet.** Confirmed via repo-wide grep — only mentioned in `Skeleton.tsx:45` (doc comment) and `docs/A5 · Performance.md` (illustrative, non-authoritative snippet). Per spec's own resolution and MAP's "one component per file, reusable" convention, this research recommends placing it at `app/core/components/JobCard.tsx` (new file) and exporting it from `app/core/components/index.ts`, since it is plausibly reused inside the future detail sheet — but this is a plan-level decision, not a blocking ambiguity.
- **`Select`'s `value: T` prop is non-optional** (`Select.tsx:13`), but `JobFilters.city`/`JobFilters.sort` fields are optional/defaulted in the schema. The plan needs an explicit sentinel value (e.g. `''` for "all cities") for the city `Select` since `city` has no schema default, unlike `sort` (which defaults to `'date_desc'` and so always has a concrete value to pass).
- **Salary input debounce/trigger mechanism (R3)** is explicitly left to the plan by the spec. Both `useDebounce` (reusing the exact same primitive as search, `useDebounce.ts:10`) and on-blur are viable with existing components (`Input` exposes `onBlur` pass-through, `Input.tsx:12,46-49`); no codebase precedent favors one over the other — this is a plan-level call, not an ambiguity requiring escalation.
- **No existing precedent for a `useEffect`-driven "debounced value changed → setFilters + refetch"** wiring in the codebase (the pattern exists in isolation via `useDebounce` and `useJobs`/`jobs.store` separately, but nothing composes them yet). Not a blocker — both primitives are independently well-defined and typed.

No ambiguity in this research rises to the level of a blocking **AMBIGUITY ESCALATION**; the two items flagged as open in the spec (FlashList API drift, `JobCard` location) are resolved above from source rather than escalated.

## Handoff to PLAN

- State/actions to wire: `useJobsStore` for `jobs, filters, pagination, isLoading, error, activeJobId, activeJobIndex` (`app/store/jobs.store.ts:4-19`) + `useJobs()` for `fetchPage/fetchNextPage/refetch` (`app/core/hooks/useJobs.ts:5-9`). Remember `refetch()` already resets the list internally (`useJobs.ts:48-51`) — never call `resetList()` from the screen directly.
- Build `JobCard` as a new file `app/core/components/JobCard.tsx`, exported via `app/core/components/index.ts`, composed from `Card` + `Badge` + `useTheme()`; memoize per `docs/A5 · Performance.md:85-90`'s pattern comparing `job.id`. On press: call `useJobsStore.getState().setActiveJob(job.id, index)`.
- Use `FlashList` from the installed `@shopify/flash-list@2.3.2` (v2, JS-only, no native estimate needed): pass `data`, `renderItem`, `keyExtractor={(item) => item.id}`, `getItemType={(item) => item.salary ? 'with-salary' : 'without-salary'}`, `drawDistance={250}`, `onEndReached={fetchNextPage}`, `onEndReachedThreshold` (default `0.5` is fine unless the plan wants tighter). **Do not** pass `estimatedItemSize` — it does not exist on this version's props and A5's doc is stale on that one point only.
- Filter/search UI: `Input` (search bar, debounced via `useDebounce(value, 300)`) and two `Input`s with `keyboardType="numeric"` for `salary_min`/`salary_max`; `Select<T>` for city and for the 5-value sort enum (`SelectOption<T>` per `Select.tsx:6-9`) — city needs a sentinel "all" option since it's optional with no schema default, unlike `sort` which defaults to `'date_desc'`.
- Loading/error/empty branches map directly to existing components with no new UI primitives needed: `isLoading && jobs.length === 0` → `JobCardSkeleton` (`Skeleton.tsx:46`); `error && jobs.length === 0` → `ErrorState` with `onRetry={refetch}` (`ErrorState.tsx:5-23`); `!isLoading && !error && jobs.length === 0` → `EmptyState` (`EmptyState.tsx:7-56`).
- Replace `app/app/(protected)/(tabs)/index.tsx:1-31`'s placeholder wholesale, keeping the default-export shape Expo Router expects for the `index` route already wired with title `'Búsqueda'` (`(tabs)/_layout.tsx:23`).
