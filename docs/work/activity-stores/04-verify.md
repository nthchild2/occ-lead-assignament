## Findings

- **`add`/`remove` re-throw after rollback (both stores), unlike `auth.store.ts`'s `logout()` which swallows.** This is intentional and consistent between `applications.store.ts` and `favorites.store.ts` (identical `catch (error) { ...rollback...; throw error }` shape in both). The plan explicitly specifies this in change #3/#4's description ("the caught error is re-thrown after rollback so a caller ... can react, matching R5's 'rolling back on failure' wording without swallowing the error silently") and flags the `logout()` divergence explicitly in "Risks & rollback" as an intentional, justified choice — not an oversight. `logout()`'s swallow is itself intentional for a different reason (local logout must always succeed regardless of network state, per its own comment). The two behaviors solve different problems; consistency is correctly scoped to "both new stores agree with each other," not "matches every existing store unconditionally."
  - Tests correctly reflect the re-throw: both rollback tests use `await expect(useXStore.getState().add/remove(...)).rejects.toThrow()`, not a silent-catch assumption. Confirmed by direct read of both `*.store.test.ts` files.
- **No `any`/`!` in any of the 10 new files** — grepped directly, none found.
- **No `fetch`/`axios` outside the two new services** — grepped `app/store/applications.store.ts`, `app/store/favorites.store.ts`, `app/core/lib/activityStatus.ts`; none found. Both stores go through `applicationsService`/`favoritesService` exclusively.
- **Local schema duplication** (`MessageResponseSchema` defined separately in both service files) is as planned and matches the `LogoutResponseSchema` precedent in `auth.service.ts`; not a defect.
- Minor process note (not a defect in the shipped code): see "Process note" above regarding a transient mutation artifact caught in a commit and corrected during this verification run.
