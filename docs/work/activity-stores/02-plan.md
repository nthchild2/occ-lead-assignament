## Approach

Build two thin service modules (`applications.service.ts`, `favorites.service.ts`) over `be-apply-fav`'s six endpoints, then two plain, non-persisted Zustand stores (`applications.store.ts`, `favorites.store.ts`) that own `fetch()`/`add()`/`remove()`/`reset()` directly on the store — mirroring `auth.store.ts`'s "store action calls the service inline" precedent, not `jobs.store.ts` + `useJobs.ts`'s "hook owns fetch, store is a dumb reducer" split. Spec R5 explicitly requires `fetch`/`add`/`remove` **on the store**, which settles that tension in `auth.store.ts`'s favor; `jobs.store.ts` is read-only pattern reference, never touched. `add`/`remove` are optimistic: mutate `items` synchronously, then call the service, rolling back to the pre-mutation state on failure. A new `app/core/lib/activityStatus.ts` (new directory) provides `isJobApplied`/`isJobFavorited` as plain functions reading `getState()` at call time, so they're usable outside React. Two endpoints (`POST apply`, and the three message-only mutation responses) have no `@occ/shared` schema; per the `LogoutResponseSchema` precedent in `auth.service.ts:7-10`, small local Zod schemas are defined inside the two new service files rather than editing `packages/shared`. Rejected alternative: putting fetch/mutate orchestration in new `useApplications`/`useFavorites` hooks (the `useJobs.ts` pattern) — rejected because R5 names `fetch`/`add`/`remove` as actions "each store exposes," and mirroring two different orchestration patterns across four domain stores would be an inconsistency with no requirement backing it.

Every rollback test seeds a distinguishable prior state (a named item present before the mutation), mocks a failing HTTP response for the mutating call, and asserts exact (`toEqual`) equality of `items` against the pre-mutation array afterward — not merely a length or non-null check — so a regression that fails to roll back (or rolls back the wrong item) will fail the test.

## Risks & rollback

- **Divergent orchestration pattern vs. `jobs.store`/`useJobs.ts`.** Intentional, per spec R5 and research's resolution — documented in Approach so a future reader doesn't "fix" it back to the hook pattern. If review disagrees, the fix is confined to changes #3/#4 (no service or lib changes needed).
- **Re-throwing after rollback in `add`/`remove`.** Not explicitly required by the spec text, but needed so a future caller (`job-detail-sheet`, out of scope here) can distinguish "optimistic UI reverted, please show an error" from "silently succeeded." If a verify-phase reviewer decides this diverges from precedent (`auth.store.ts`'s `logout()` deliberately swallows its error), the change is confined to removing the `throw err` line in changes #3/#4 and adjusting the two rollback test assertions (#8d/f, #9d/f) from `.rejects.toThrow()` to a resolved-promise check — no service or lib file is affected.
- **Local schema duplication** (`MessageResponseSchema` defined separately in both `applications.service.ts` and `favorites.service.ts`) is intentional, matching the existing one-off-per-file precedent (`auth.service.ts:7-10`) rather than introducing a new shared `core/services/` schema-util module not asked for by any requirement. If this is flagged in review as unwanted duplication, it collapses to a single shared local constant with no change to store/lib files.
- **Rollback failure risk:** if a service call throws something other than `ApiError` (e.g. a schema-parse `ZodError` on a malformed success response), the store's `catch` block still fires and still rolls back — the catch is untyped/generic (`catch (err)`), not narrowed to `ApiError`, so this is covered by construction, not a gap.
- **If verify fails on the local-schema-vs-shared-schema decision** (e.g. reviewer wants `ApplyResponseSchema` promoted to `@occ/shared`), rollback is scoped to changes #1/#2 only — stores and lib import types (`Application`, `Job`), not the local response schemas, so no downstream file needs to change.
