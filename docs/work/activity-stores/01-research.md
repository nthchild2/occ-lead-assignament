## Existing patterns to follow

- **Zustand store (non-persisted, list-shaped)** → follow `app/store/jobs.store.ts:41-70`; `create<T>((set, get) => ({...}))`, no `persist` middleware (A2 Decision 1). `applications.store.ts`/`favorites.store.ts` are new siblings of the same shape.
- **Store action calling the service directly** (not a hook) → follow `app/store/auth.store.ts:23-47` (`login`, `logout`, `hydrate` each call `authService.*` inline, then `set(...)`). This is the correct precedent for R5's `fetch()`/`add()`/`remove()` living **on the store**, since R5 is explicit that the store itself exposes these actions (unlike `jobs.store`, where fetch orchestration was pulled into `useJobs.ts`).
- **Thin service module** → follow `app/core/services/auth.service.ts:1-23` or `jobs.service.ts:23-25`: one exported function per endpoint, each a one-line call to `get`/`post`/`del` with a schema; no branching logic in the service.
- **Local schema for an endpoint not covered by `@occ/shared`** → follow `app/core/services/auth.service.ts:7-10` (`LogoutResponseSchema = z.object({ data: z.unknown() })`, defined locally, not added to the shared package). Needed here for: `POST /jobs/:id/apply`'s `{ data: Application }` response (use `z.object({ data: ApplicationSchema })`, built from the existing `ApplicationSchema` import) and the message-envelope responses on `DELETE apply` / `POST favorite` / `DELETE favorite` (`z.object({ data: z.object({ message: z.string() }) })`).
- **Store test convention (plain store, no hook)** → follow `app/store/auth.store.test.ts:1-115` or `app/store/jobs.store.test.ts:1-102`: mock `global.fetch`, call `useXStore.getState().actionName(...)` directly (no `renderHook`/`@testing-library/react-native`), assert via `getState()` after `await`. `renderHook` is reserved for hooks (`useJobs.test.ts`), not needed for this feature since R5 puts the async actions on the stores themselves.
- **Optimistic mutation + rollback** — no existing store in this repo yet does optimistic add/remove-with-rollback; there is no direct precedent to cite for that specific mechanic. The nearest analog is `auth.store.ts:37-39`'s `clearSession()` as a small, isolated state-reset action, and `jobs.store.ts:55-57`'s `resetList()` reset-to-initial pattern — both establish the "one focused setter" style to keep the optimistic/rollback logic in, but the rollback control flow itself must be authored fresh in the plan.
- **`core/lib/` pure utility module** → no existing file yet (directory does not exist — confirmed below); `docs/A1 · Monorepo Architecture.md:33` documents it as the destination for "pure utilities (formatters, validators)." `activityStatus.ts` (or similar) reading `useApplicationsStore.getState()` / `useFavoritesStore.getState()` would be the first file here. Since it only reads store `getState()` (no JSX, no `app/app/` import), it does not trip `.eslintrc.js:33-44`'s `core/` → `app/` restriction — that rule only blocks importing from `app/app/`, not from `app/store/`.

## Constraints that apply

- `core/` must not import from `app/` — `.eslintrc.js:33-44` (rationale: A1 Decision 2). Stores live in `app/store/`, so this only bounds `app/core/lib/activityStatus.ts` (fine, it imports stores, not `app/app/`) and the two new services (fine, services import `api.ts` and `@occ/shared`, not `app/app/`).
- No `fetch`/`axios` in components or stores — use `core/services/` (`docs/MAP.md:72`). The stores must call `applications.service`/`favorites.service`, never `fetch`/`api.ts` directly.
- One store per domain; only `auth.store` is persisted, all others reset on launch — `docs/MAP.md:45` / A2 Decision 1 (`docs/A2 · State & Data Strategy.md:40`). No `persist` middleware on either new store.
- No `any`; API types via `z.infer<>` — `.eslintrc.js` (per `docs/MAP.md:71`). Response/request types must derive from Zod schemas (shared or locally defined per the `LogoutResponseSchema` pattern), not hand-written interfaces.
- App tests are co-located (`*.test.ts` next to the unit) — `docs/MAP.md:59`. `applications.store.test.ts`, `favorites.store.test.ts`, `applications.service.test.ts` (if added), `favorites.service.test.ts` (if added), `activityStatus.test.ts` all sit beside their source files.
- Mock `global.fetch` or the service module in app tests — do **not** use msw — `docs/MAP.md:61`.
- TypeScript strict mode repo-wide (`tsconfig.base.json`, per `docs/MAP.md:75`).

## Risks & unknowns

- **Two competing "where does fetch orchestration live" precedents exist in the codebase**: `jobs.store.ts` + `useJobs.ts` keep fetch/error-handling in a hook that calls store setters, while `auth.store.ts` keeps it directly in store actions. Spec R5 is explicit ("Each store exposes `fetch()` ... `add`/`remove`") that these live **on the store**, which resolves this in favor of the `auth.store.ts` precedent — not an ambiguity, but the plan should note explicitly _why_ it diverges from the more recent `jobs.store`/`useJobs` split, so a future reader doesn't assume it's inconsistent by accident.
- **Optimistic-add payload shape (spec's own pre-flagged open question, resolved here):** Confirmed from `applications.service.ts:43-58` that `appliedAt` is generated server-side (`new Date().toISOString()`), and the router (`applications.router.ts:34-42`) returns the full `Application` object (`{ jobId, appliedAt, job }`) on success — so the service call itself returns everything needed. For **synchronous optimism** (item must appear before the network response resolves), the store's `add(job)` action should synthesize a client-side `appliedAt` (`new Date().toISOString()`) into a local `Application` shape immediately, then on the service's real response, the item can optionally be reconciled/replaced with the server-authoritative object (or left as-is until the next `fetch()`, per spec's own suggested resolution). `favorites.store`'s `add(job)` takes the full `Job` directly (no synthesized field needed — `Job` has no `favoritedAt`), simpler and directly satisfies R6's cross-reference-by-id need. This is a plan-level decision, not a blocking ambiguity — the data needed to decide is now on record.
- **`isJobApplied`/`isJobFavorited` location (spec's own pre-flagged open question, resolved here):** `app/core/lib/` does not yet exist but is the documented convention (`docs/A1 · Monorepo Architecture.md:33`, "pure utilities"). `docs/MAP.md`'s own "Entry points" table (`docs/MAP.md:16`) also lists `core/` generically as "Components, hooks, theme, services, lib." A new file `app/core/lib/activityStatus.ts` (or `isJobApplied.ts`/`activityStatus.ts` — naming is a plan-level call) reading `useApplicationsStore.getState().items` / `useFavoritesStore.getState().items` and matching by `job.id`/`jobId` is consistent with both the eslint boundary (`core` importing `app/store/*` is not restricted; only `core` → `app/app/*` is) and the "pure utility, no fetch" convention. Not a blocking ambiguity.
- **No shared schema exists for the apply/cancel/favorite/unfavorite mutation response bodies.** Confirmed by reading the actual routers: `POST apply` returns `{ data: Application }` (an unwrapped object, not the `{ data: { items: [...] } }` shape `ApplicationListResponseSchema` expects), and `DELETE apply` / `POST favorite` / `DELETE favorite` all return `{ data: { message: string } }`. Neither shape has a pre-built `@occ/shared` schema. The plan must decide, per the `LogoutResponseSchema` precedent, whether to define small local schemas in the two new service files (recommended, matches precedent) or parse loosely with `z.unknown()`/skip strict parsing for the mutation calls (the service still needs _some_ schema argument since `post`/`del` in `api.ts:110-151` require one).

## Handoff to PLAN

- Mirror `auth.store.ts` (not `jobs.store.ts`/`useJobs.ts`) for where `fetch()`/`add()`/`remove()` live — store actions call the service directly; test with `getState()`, no `renderHook`, per `auth.store.test.ts`.
- Real routes (confirmed from routers + `app.ts:35-40`): `POST/DELETE /jobs/:id/apply`, `POST/DELETE /jobs/:id/favorite`, `GET /applications`, `GET /favorites` — all `authMiddleware`-protected, so every service call passes `auth=true` (the `get`/`post`/`del` default).
- Response shapes: `GET /applications` → `ApplicationListResponseSchema`; `GET /favorites` → `FavoriteListResponseSchema`; `POST apply` → `{ data: Application }` (needs a local `z.object({ data: ApplicationSchema })`, no shared schema exists); `DELETE apply`/`POST favorite`/`DELETE favorite` → `{ data: { message: string } }` (needs a local message schema, following the `LogoutResponseSchema` precedent in `auth.service.ts:7-10`).
- New files: `app/store/applications.store.ts`, `app/store/favorites.store.ts`, `app/core/services/applications.service.ts`, `app/core/services/favorites.service.ts`, `app/core/lib/activityStatus.ts` (directory `app/core/lib/` does not exist yet — this creates it), plus co-located `*.test.ts` for each.
- Optimistic add: applications need a client-synthesized `appliedAt` (server always overwrites/matches on real success, per `applications.service.ts:53`); favorites' `add(job)` takes the full `Job` directly, no synthesis needed. Rollback on failure = remove the optimistically-added item / re-add the optimistically-removed item.
- R7's `reset()` is a primitive only — no logout wiring in this ticket (confirmed out of scope both in spec and in `docs/A2 · State & Data Strategy.md:237-240`'s diagram, which shows the reset happening as part of a larger logout flow this ticket doesn't own).
